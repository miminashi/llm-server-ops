## 前提・目的

前身レポート Phase D で `numactl --cpunodebind=1 --membind=1 -- + --threads 40` (C-D3) が **eval 15.03 t/s（C-C2 比 +26.2%）** で新たに採用された。C-D3 がなぜ速いのかの 3 要因仮説のうち、以下 3 件が未検証だった:

- **G-3**: `--numa isolate`（llama.cpp フラグ）を numactl と併用した場合の挙動確認

副次的に以下 1 件:

- **G-4**: `--threads 60` での中間値（40 CPU に対して 60 スレッド）→ オーバーサブスクリプション境界の再確認

これらを検証し、C-D3 を上回る構成があれば採用候補とすることを目的とした。

## 計測手順

各構成とも共通プロトコル（Phase D と同一）:

1. `stop.sh t120h-p100` で既存プロセス停止 → 15 秒待機（`ps` で停止確認）
2. `start_phaseE.sh <variant>` で起動、`/health` 200 確認
3. llama-server 本体 PID を `ps -eo pid,comm,args | awk '$2=="llama-server" {print $1; exit}'` で取得
4. `measure_phaseE.sh <pid> <tag>` 実行
5. 3 run 実施: 各 run 前 60 秒 cooldown、eval プロンプト `"Write a short haiku about autumn."`、`max_tokens=256`、`stream=false`
6. curl レスポンスの `timings.predicted_per_second` / `timings.prompt_per_second` を記録
7. `numastat -p $PID` を Run 1 直前と Run 3 終了後に取得
8. `nvidia-smi dmon` 20 秒、Run 3 のみ `/proc/$PID/status` スナップショット

## ボトルネック分析

### C-E2（threads=20, SMT なし）が C-E1（threads=40, SMT 込み）と同等以上

- C-E1: 14.27 t/s / C-E2: 14.54 t/s → **threads を半分に減らしても 1.9% 向上**
- Threads=106 vs 126（20 スレッド減）、voluntary_ctxt_switches=930 vs 3,300（71.8% 減）
- **Phase D で提示した「SMT 活用」仮説は支持されない**: SMT なし（物理コア 20 のみ）で同等の性能が出ており、SMT による追加スレッドは性能に寄与していない
- CPU 側 FFN 計算は 20 物理コアで既に飽和していると推定。HT で論理 20 スレッドを追加しても、実行ユニット競合で差は生まれない

### C-E3（threads=60）が激しく劣化（−49.3%）

- Cpus_allowed_list=20-39,60-79（Node 1 の 40 CPU）に対し 60 スレッドを押し込む → **20 スレッド分のオーバーサブスクリプション**
- 7.62 t/s は Phase B の `-N1 -m1 + threads=-1`（実質 80 スレッド）の 11.50 t/s より更に遅い。これは次の理由による:
  - Phase B は 80 スレッドを Cpus_allowed_list=0-79 全体に分散 → CPU あたり 1 スレッド
  - C-E3 は 60 スレッドを 40 CPU に押し込む → 特定 CPU に 1.5 スレッド
- 「2:1 スケール寄りのオーバーサブスクリプション」は「1:1 均等分散」より致命的であることが判明

### C-E4（taskset 併用）が C-E1 とほぼ同等（+0.8%）

- C-E1 の `numactl --cpunodebind=1` と C-E4 の `taskset -c 20-39,60-79` は **実効的に同一の CPU 制約**
- voluntary_ctxt_switches の比較は取れなかったが、Cpus_allowed_list=20-39,60-79 で同じ
- **numactl と taskset の差はない**。Linux スケジューラが既に十分な局所性を提供

### C-E5（--numa isolate 併用）が意外に +5.1%（初回）

- 本計測で最も興味深い結果。初回 C-E5 = 15.00 t/s（C-E1 比 +5.1%）、再計測 C-E5b = 14.75 t/s（+3.4%）
- **`--numa isolate` は numactl の `--cpunodebind=1` を上書き**（Cpus_allowed_list=0-79）
- メモリは Node 1 に固定 → Node 0 スレッドは全てリモートアクセス → 本来は C-D6 相当（10.70 t/s）まで落ちるはず
- にもかかわらず同等以上の速度が出た理由は不明。仮説:
  - llama.cpp の `--numa isolate` が OpenMP thread affinity を内部で調整し、リモートアクセスを最小化する分岐割当を実装している可能性
  - 40 スレッドが 80 CPU に配置されるとき、Linux スケジューラが Node 1 側を優先する（メモリ局所性を考慮）
  - eval 時のホットパスが CPU 側 FFN 計算に偏っており、cold キャッシュ層は最低限しか触らないため、リモートアクセス影響が小さい
- ただし計測揺らぎ（E5: 15.00 / E5b: 14.75）が ~0.25 t/s あるため、Phase D の C-D3 (15.03) との差は **統計的有意でない**

### C-E1 が Phase D の C-D3 (15.03) を再現できなかった（14.27）

- 同一サーバ・同一プロセス・同一構成なのに **0.76 t/s 劣化**
- 稼働時間の経過（Phase D 計測から約 1 時間後）で以下のいずれか:
  - メモリ断片化の進行
  - Node 1 内の CPU バインドが徐々に崩れた（voluntary_ctxt_switches が 273 → 3,300 に増加）
  - 他プロセスによる キャッシュ汚染
- **結論**: C-D3 採用構成は長時間稼働で若干劣化する可能性あり。定期再起動が有効か要検証

## 採用判定

| 構成 | 中央値 | C-D3 比 | 判定基準 +3% | 判定 |
|------|------:|--------:|:-----------:|:----:|
| **C-D3 (Phase D 採用、維持)** | 15.03 | 基準 | ― | **継続採用** |
| C-E5（`--numa isolate` 併用） | 15.00 / 14.75 | −0.2% / −1.9% | 未達成 | 非採用（要追試） |
| C-E1（C-D3 追試） | 14.27 | −5.1% | ― | 継続劣化の観測 |
| C-E4（taskset） | 14.38 | −4.3% | 未達成 | 非採用 |
| C-E2（threads=20） | 14.54 | −3.3% | 未達成 | 非採用（同等、SMT 無寄与の証拠） |
| C-E3（threads=60） | 7.62 | −49.3% | 論外 | 非採用 |

**新採用なし。C-D3 (`numactl --cpunodebind=1 --membind=1 -- + --threads 40`) を継続採用。**

ただし **Phase D の 3 要因説「NUMA 局所 + 1:1 マッピング + SMT 活用」のうち SMT 活用は否定**された。本質は以下 2 要因に整理される:

1. **メモリ局所性**: `--membind=1` でモデル重みへの Node 内アクセスを確保
2. **非オーバーサブスクリプション**: `--threads ≤ 利用可能物理コア数`（20〜40）で ctxt_switches を抑制

SMT は「あっても悪くない」程度で、省電力 / 他プロセス共存を重視するなら `--threads 20` でも実用上同等。
