## 前提・目的

Phase R-ctx3 レポート末尾「未検証事項 / 新規項目」および「検証完了後に実施すべき TODO / 新規項目」両方で最優先登録:

> **CUDA0 二次モデルの ub 依存性**: 本 Phase は ub=2048 固定、ub 可変で二次項係数 c が変化するかは未検証。`CUDA0 = a(ub) + b(ub)·Δctx + c(ub)·Δctx²` の 3 つの係数が `ub` にどう依存するかの追加計測が必要 (Phase R-ctx3-ub 候補)

Phase R-ctx3 で確定した compute buffer 式:
- `CUDA0     = 1046.29 + 3.269e-3·Δctx + 5.770e-8·Δctx² + 0.077·(ub-2048)` (Δctx=ctx-16384, R²=0.99998)
- `CUDA1/2   = 520.06  + 0.003906·Δctx + 0.254·(ub-2048)` (R²=1.00000000)
- `CUDA3     = 0.9824·ub` (ctx 完全不依存、4 点で偏差 0.000 MiB)
- `CUDA_Host = 176.08  + 0.007812·Δctx + 0.086·(ub-2048)` (R²=1.00000000)

上記はすべて「ub=2048 固定の ctx fit」に「Phase Q 由来の ub 線形補正」を後付けした**近似**であり、ub × ctx 相互作用は検証していない。本 Phase S は ub=512/1024/4096/8192 × ctx=32k/65k の 8 条件を新規計測し、既存 8 点 (Phase Q 5 点 + Phase R-ctx3 2 点 + Phase R 1 点) と合わせて**合計 16 点で 2 軸最小二乗フィット**を実施することで:

1. **CUDA0 の二次モデルが ub × ctx 相互作用項を含むかどうかを実証**
2. **CUDA1/2 / CUDA_Host の ub × ctx 相互作用項の有無を 16 点で実証**
3. 起動前 lint の予測誤差を ub 可変時でも ≤ 1% に抑えられる 2 軸モデルを確定
4. 副次目的: ub 可変による eval/prompt 性能の ctx 依存性を把握

## ボトルネック・副次発見の分析

### 1. CUDA1 / CUDA2 / CUDA_Host の ub × ctx 相互作用項 — 新発見

3 params 線形モデル (相互作用項なし) では CUDA1/2 で R²=0.989、Host で R²=0.922 と目標未達。**4 params モデル (+e·Δctx·Δub) を導入することで両者とも R² ≥ 0.99999**。

物理解釈: **attention score staging buffer が各 GPU に O(ub × ctx) で配置される**。fa=1 (FlashAttention) でも内部タイルの部分スコアを一時的に staging する必要があり、そのサイズが ub と ctx の積に比例する。CUDA_Host は CPU ↔ GPU 双方向のコピーバッファを持つため、cross_e=3.815e-6 が CUDA1/2 の 1.910e-6 のちょうど 2 倍となる (1 方向分の 2 倍)。

確定 4 params モデル（ub=128〜8192 × ctx=16k〜131k で実証、Phase S 16 点フィット）:
```
Δctx = ctx - 16384, Δub = ub - 2048
CUDA1/2   = 520.26 + 3.903e-3·Δctx + 0.2538·Δub + 1.910e-6·Δctx·Δub   [R²=0.99999965]
CUDA_Host = 176.08 + 7.813e-3·Δctx + 0.0860·Δub + 3.815e-6·Δctx·Δub   [R²=1.00000000]
```

### 2. CUDA0 の区分的挙動 (ub=2048 閾値) — 新発見

CUDA0 は **ub ≤ 1024 でほぼ ctx/ub 独立平坦 (961-973 MiB)**、**ub ≥ 2048 で急増**。この区分性は単一多項式フィットでは R² ≤ 0.99 の壁を超えられない本質的理由。

ctx=16k/32k/65k の ub=512/1024 行はすべて CUDA0 ≈ 966.50 / 973.00 と完全同一値。これは llama.cpp scheduler が「ub ≤ 1024 では CUDA0 に追加 staging を行わず、ub ≥ 2048 で staging 開始」という**閾値判定ロジック**を持つことを強く示唆。

実用予測モデルとして**区分的に運用**する方が精度が高い:

```
CUDA0 = 966.50 + 0.0064·ub              [ub ≤ 1024, 全 ctx 共通]
CUDA0 ≈ 6p fit (下記)                    [ub ≥ 2048]
      ≈ 1116.34 + 4.996e-3·Δctx + 3.670e-8·Δctx² + 0.1115·Δub + 6.016e-6·Δctx·Δub + 9.104e-6·Δub²
```

運用では保守的に **max(ub ≤ 1024 式, 6p fit) + 300 MiB マージン**を取ることで、max_err 236 MiB を吸収可能。

### 3. CUDA3 の純 ub 比例性 — 16 点で数学的確定

Phase R-ctx3 では 4 点で偏差 0.000 MiB と報告したが、本 Phase では **16 点すべてで偏差 0.000 MiB**。切片ゼロ強制の純比例フィット `g·ub` で傾き 0.982422、R²=1.00000000。

**物理的意味**: CUDA3 は純粋に attention kernel の計算バッファで、層 28-30 (GPU3 担当層) の attention score / softmax 出力を保持するための O(ub × head_dim × n_head × layers) buffer。ctx に完全不依存なのは fa=1 によって KV cache が別 buffer (96 · ctx/16384 MiB) に分離されているため。これが実験的・物理的・数学的・統計的に 16 点で確定。

### 4. Phase R-ctx3 単変量モデルの破綻範囲

Phase R-ctx3 モデルは以下の範囲で誤差 ≤ 5% (実用許容):
- **ub ∈ [512, 2048] × ctx = 16384** 付近（Phase Q 元データ域）
- **ub = 2048 × ctx ∈ [16384, 131072]**（Phase R-ctx3 元データ域）

破綻範囲 (誤差 > 10%):
- ub ∈ {512, 1024} × ctx ∈ {32k, 65k}: Host で -29〜-67% (過小予測)
- ub ∈ {4096, 8192} × ctx ∈ {32k, 65k}: CUDA0 で +50〜+137%, Host で +26〜+106% (過大予測)

これは「ub ≠ 2048 では Phase R-ctx3 モデルで lint しない」というルールを暗示。新 Phase S 4p (CUDA1/2/Host) / 6p (CUDA0) モデルで代替する必要がある。

### 5. eval 性能の ub・ctx 2 軸挙動

ctx=65k × ub=512 × 1k prompt で **15.39 t/s の本 Phase 最速値**。ctx=16k (Phase Q P1 15.42 t/s) と同程度で、ctx=131k (Phase R 14.89 t/s) より +3.3% 高速。

観察: **eval は ub が小さいほど速い傾向**。ただし ctx=32k × ub=2048 (Phase R-ctx3 15.06) と ctx=32k × ub=512 (本 Phase 14.64) では ub=2048 の方が速いこともあり、ub=2048 付近に eval 最適値が存在する可能性。Phase Q-3 候補 (ub=1536/1792/2304/2560/2816/3072 の微細スキャン) で確認必要。

prompt 速度は ub に比例 (5.3× for 16× ub 増加)。prompt-bound ワークロードでは ub=8192 が最適。

### 6. batch_remaining.sh の `ssh -f` パイプ詰まり — 運用教訓

S2 段階で `bash start_phaseS.sh 2>&1 | tail -10` のパイプが 26 分間停止した原因:
- start_phaseS.sh 内の `ssh -f "$HOST" "cd ~/llama.cpp && nohup bash -c \"...\" > log 2>&1 < /dev/null &"`
- `ssh -f` は detach 後もマスタープロセスを保持し、stdout が parent の pipe に残る
- 親の tail -10 は EOF を受け取れず hang

**回避策 (batch_S3onwards.sh で実装済み)**:
- `start_phaseS.sh > start_stdout_ctx${CTX}_ub${UB}.log 2>&1 &` で file redirect + background
- `curl -m 5` で timeout 付き health チェックで同期待機
- ssh 接続には `-o ConnectTimeout=10 -o BatchMode=yes` で hang 抑制

この改善は、今後の長時間バッチ計測 Phase に導入すべきパターン。

## 補足

### Phase S の核心発見

1. **CUDA3 = 0.9824·ub は 16 点で max_err 0.000 MiB** — 純 ub 比例を ctx × ub 2 軸 16 点で数学的・実験的に確定
2. **CUDA1/2 / CUDA_Host は 4 params 2 軸モデル (cross 項 e 必須)** — 3 params 線形では R²=0.92-0.99 と不十分、4 params で R² ≥ 0.99999 に改善
3. **CUDA_Host の cross_e 係数は CUDA1/2 の 2 倍** — 双方向コピーバッファとしての物理的整合性
4. **CUDA0 は ub=2048 閾値の区分的挙動** — ub ≤ 1024 で ctx 独立平坦、ub ≥ 2048 で急増。単一多項式 fit で R² 0.99 の壁
5. **Phase R-ctx3 単変量モデルは ub ≠ 2048 で最大 +137% 誤差** — ub=2048 固定前提のため、運用では 2 軸モデルへの置換必須
6. **eval 最速**: ctx=65k × ub=512 × 1k prompt で **15.39 t/s** (Phase R-ctx3 の 15.32 を上回る本 Phase 最速)
7. **prompt 速度は ub に線形比例** — ub=512 → 8192 で 34.8 → 184.5 t/s (5.3×)
8. **graph 構造 (4473 nodes, splits=136+77) は 16 点で完全不変** — ctx・ub 両軸に独立

  KV/GPU = 96 × (ctx / 16384) MiB                                               [16 点 max_err 0.000]
