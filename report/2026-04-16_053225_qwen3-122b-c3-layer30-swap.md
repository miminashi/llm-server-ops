## 前提・目的

- **目的**: C-1 構成（layer 14-19 を GPU 復帰、eval 12.06 t/s）の拡張として、layer 24 を避けて **layer 25-30** を GPU に追加復帰させ、CUDA1 のセーフティマージンを維持しつつ CUDA2 の空き VRAM を活用する
- **仮説**: `--split-mode layer` の既定配置では layer 24 が CUDA1/CUDA2 境界の CUDA1 側にアライメントされており、C-2 (14-19 + 24-29) では layer 24 が CUDA1 を圧迫して空きが 2098 → 706 MiB に縮小した。layer 24 を CPU に戻し layer 30 を GPU に追加すれば、**CUDA1 の負荷は C-1 のまま** 6 層分の expert を CUDA2 に集中搭載できる
- **成功条件**: ctx 131072 で OOM なく稼働、全 GPU 空き ≥ 2 GiB（特に CUDA1 ≥ 1500 MiB）、eval が C-1 と同等以上

## 環境情報

## 実験構成

### `-ot` パターン

```
blk\.([0-9]|1[0-3]|2[0-4]|3[1-9]|4[0-7])\.ffn_.*_exps\.weight=CPU
```

- **CPU 層**: 0-13, 20-24, 31-47（計 36 層）
- **GPU 層**: **14-19, 25-30**（計 12 層、C-1 の倍）

他起動パラメータは C-1 と完全一致（`-ctx-size 131072 -b 8192 -ub 8192 --flash-attn 1 --cache-type-k q8_0 --cache-type-v q8_0`）。

## 詳細

### C-3 VRAM 配分（load_tensors + KV + RS + compute）

ログ実測値（MiB 単位）:

| GPU | model | KV | RS | compute | 合計 | nvidia-smi used | free |
|-----|-------|-----|-----|---------|------|-----------------|------|
| CUDA0 | 1301 | 408 | 41 | 7648 | 9398 | 9703 | **6568** |
| CUDA1 | **9551** | 408 | 37 | 3872 | 13868 | 14173 | **2098** |
| CUDA2 | **9551** | 408 | 37 | 3872 | 13868 | 14173 | **2098** |
| CUDA3 | 1693 | 408 | 33 | 8048 | 10182 | 10485 | **5786** |

- **CUDA1 model**: 9551 MiB（C-1 と完全一致） → layer 14-19 のみが CUDA1 に搭載されたことを示す
- **CUDA2 model**: 9551 MiB（C-1 の 1199 から +8352 MiB = 6 層分） → layer 25-30 がすべて CUDA2 に搭載
- 対称配置（CUDA1 と CUDA2 が同量 9551 MiB）が実現

### 仮説判定

| 仮説 | 結果 | 根拠 |
|------|------|------|
| **layer 24 が CUDA1 境界にある** | **成立** | layer 24 を除き layer 25-30 のみ GPU 指定すると、CUDA1 model が C-1 と完全同一 (9551 MiB) のままだった |
| **layer 30 は CUDA2 に配置される** | **成立** | CUDA2 model が +8352 MiB = 6 層分増加（layer 25-30 すべて CUDA2） |
| **eval が C-1 より向上** | **弱成立** | 中央値 12.19 t/s で C-1 再測定比 +2.8%、原典比 +1.1%。C-2 (12.29) との差は 0.8% で試行ばらつき範囲 |
| **eval +5〜10% の期待値** | **不成立** | GPU 層を倍増 (6→12) しても eval 向上は小幅。ボトルネックは残り 36 層の CPU 側 expert 計算にあると推定 |

### `--split-mode layer` の層→GPU アライメント（実測から判明）

本実験 + C-2/C-2' の差分実験から、P100×4 環境での 48 層配置について以下が推定できる:

| GPU | 担当層範囲（推定） | 備考 |
|-----|-------------------|------|
| CUDA0 | embedding + 層 0-13 付近 | compute buffer が大（7648 MiB） |
| CUDA1 | 層 14-24 付近 | layer 24 が CUDA1 末尾、layer 25 が CUDA2 先頭 |
| CUDA2 | 層 25-35 付近 | layer 30 を含む |
| CUDA3 | 層 36-47 + output 層 | compute buffer が大（8048 MiB） |

**注**: この推定は「expert 重みをどの GPU に乗せるか」の観点であり、load_tensors 本体のログには各層の GPU 割り当てが明示されないため、`-ot` の差分実験でしか推定できない。

## 結論と判断

- **仮説の前半（CUDA1 マージン維持）は完全成立**: CUDA1 の空きは C-1 と完全同一の 2098 MiB、CUDA1 model も 9551 MiB で不変
- **仮説の後半（eval +5〜10%）は不成立**: GPU 層を 6 → 12 層に倍増しても eval 向上は C-2/C-2' と同水準（+2〜4%）。ボトルネックが CPU 側 expert 計算にある可能性が高い
- **ただし C-3 を採用継続**: C-1 と同等のセーフティマージンを保ったまま GPU 利用率を倍増（6 層 → 12 層）でき、eval も C-1 と同等以上。将来の拡張（CUDA0/3 compute buffer 削減で追加余地確保など）の足場として優位
- 作業終了時点で **C-3 構成で稼働中**（ロック解放後もこの構成を継続）

  -m '$MODEL_PATH' --jinja \
  -ngl 999 -ot 'blk\\.([0-9]|1[0-3]|2[0-4]|3[1-9]|4[0-7])\\.ffn_.*_exps\\.weight=CPU' \
  --flash-attn 1 --poll 0 -b 8192 -ub 8192 \
  --n-predict 32768 --threads -1 \
  --ctx-size 131072 --parallel 1 \
  --cache-type-k q8_0 --cache-type-v q8_0 \
  --defrag-thold 0.1 \
  --temp 0.6 --top-p 0.95 --top-k 20 --min-p 0 \

  --alias 'unsloth/Qwen3.5-122B-A10B-GGUF:Q4_K_M'"

ssh -f t120h-p100 "cd ~/llama.cpp && nohup bash -c \"\$LAUNCH_CMD\" > /tmp/llama-server.log 2>&1 < /dev/null &"

until curl -sf http://10.1.4.14:8000/health > /dev/null; do sleep 5; done
