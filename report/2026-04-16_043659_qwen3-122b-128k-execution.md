## 前提・目的

- **目的**: `t120h-p100` 上の `unsloth/Qwen3.5-122B-A10B-GGUF:Q4_K_M` を **ctx 131072（128k）** で稼働させ、空き VRAM に expert 重みを戻して eval 速度を向上させる
- **段階検証方針**: プラン A（ベースライン） → B（down_proj 全層 GPU 復帰） → D（-b/-ub 4096 + B） → C-1（layer 14-19 のみ GPU 復帰）を段階的に実測し、仮説 H1/H2/H3 を判定
- **成功条件**: ctx 131072 で安定稼働、全 GPU 3 GiB 以上の空き、eval 速度がベースライン（~10 t/s）を上回る

## 環境情報

- **GPU**: NVIDIA Tesla P100-PCIE-16GB × 4（合計 64 GiB VRAM、compute capability 6.0）

## 各プラン詳細

### プラン A: ctx 131072 ベースライン

`start.sh fit 131072` で起動。既存の `-ot 'ffn_.*_exps.weight=CPU'` で全 expert 重みを CPU に置き、コンテキストだけ 128k 化した構成。

| GPU | model | KV | RS | compute | nvidia-smi | free |
|-----|-------|-----|-----|---------|-----------|------|
| CUDA0 | 1301 | 408 | 41 | 7648 | 9703 | 6568 |
| CUDA1 | 1199 | 408 | 37 | 3360 | 5309 | 10962 |
| CUDA2 | 1199 | 408 | 37 | 3360 | 5309 | 10962 |
| CUDA3 | 1693 | 408 | 33 | 8048 | 10485 | 5786 |

**eval 速度**: 10.21 t/s（prompt eval 27.46 t/s、256 tokens 生成 25.07 秒）。

### プラン B: down_proj のみ GPU 復帰（OOM）

`-ot 'ffn_(gate|up)_exps\.weight=CPU'` に変更し、down_proj 相当を GPU に戻した。

**結果**: CUDA0 で `cudaMalloc failed: out of memory` により起動失敗。allocate 試行サイズは **7552 MiB** の compute buffer。プラン A で CUDA0 が既に compute buffer 7648 MiB を占有していたため、model buffer に down_proj 分が追加されるとフィットしない。

```
ggml_backend_cuda_buffer_type_alloc_buffer: allocating 7552.00 MiB on device 0: cudaMalloc failed: out of memory
ggml_gallocr_reserve_n_impl: failed to allocate CUDA0 buffer of size 7918846080
graph_reserve: failed to allocate compute buffers
```

### プラン D: `-b 4096 -ub 4096` + B（成功、速度向上は限定的）

`-b/-ub` を半減することで compute buffer を圧縮しつつ、down_proj 全層 GPU 復帰を成立させた。

| GPU | model | KV | RS | compute | nvidia-smi | free |
|-----|-------|-----|-----|---------|-----------|------|
| CUDA0 | 8165 | 408 | 41 | 3736 | 12651 | 3620 |
| CUDA1 | 7535 | 408 | 37 | 1808 | 10089 | 6182 |
| CUDA2 | 7535 | 408 | 37 | 1808 | 10089 | 6182 |
| CUDA3 | 7501 | 408 | 33 | 4025 | 12267 | 4004 |

- compute buffer は約半分（CUDA0 7648→3736、CUDA3 8048→4025）
- model buffer は各 GPU に +6 GiB 前後追加（down_proj が 4 GPU に均等分散）
- 全 GPU 3.6 GiB 以上の空きを確保

**eval 速度**: 10.49 t/s（A 比 +2.7%）。期待値（13〜17 t/s）には届かず。

**考察**: MoE 推論では gate/up/down が同一 expert 内で連続計算されるため、down だけ GPU に戻しても gate/up の CPU→GPU データ転送がボトルネックとなり、クリティカルパスが GPU 化しない。expert 単位・層単位で丸ごと GPU に置く方が効率的と判断。

### プラン C-1: layer 14〜19 の expert を GPU に復帰（採用）

`-ot 'blk\.([0-9]|1[0-3]|2[0-9]|3[0-9]|4[0-7])\.ffn_.*_exps\.weight=CPU'`（layer 0-13, 20-47 を CPU、14-19 を GPU）。6 層分の expert（約 9 GiB）が CUDA1 に集中。

| GPU | model | KV | RS | compute | nvidia-smi | free |
|-----|-------|-----|-----|---------|-----------|------|
| CUDA0 | 1301 | 408 | 41 | 7648 | 9703 | 6568 |
| CUDA1 | **9551** | 408 | 37 | 3872 | 14173 | **2098** |
| CUDA2 | 1199 | 408 | 37 | 3360 | 5309 | 10962 |
| CUDA3 | 1693 | 408 | 33 | 8048 | 10485 | 5786 |

**eval 速度**: 12.06 t/s（A 比 **+18%**、prompt eval 31.18 t/s）。

CUDA1 の空きが 2 GiB で目標 3 GiB をやや下回るが、起動・推論ともに安定稼働を確認。

## 仮説判定

| 仮説 | 結果 | 根拠 |
|------|-----|------|
| **H1** compute buffer は ctx-size に非依存 | **部分成立** | CUDA3 は 8048→8048 MiB で変化なし。ただし CUDA0 は 4576→7648 (+3072)、CUDA1/2 は 2337→3360 (+1023) と増加 |
| **H2** compute buffer は ctx-size に弱く比例 | **部分成立** | ctx 倍化で CUDA0/1/2 の compute buffer が増加。特に CUDA0 は 67% 増と顕著。CUDA3 は既に上限張り付き |
| **H3** GPU 間不均衡は `--split-mode layer` 起因 | **成立** | CUDA0（embedding 側）と CUDA3（output 側）に compute buffer が集中。CUDA1/2 は空き豊富。layer 単位の選択的復帰（C-1）で CUDA1 にのみ expert を追加できた |

**結論**: H1/H2/H3 いずれも部分的に成立。compute buffer の ctx 依存性（H2）は限定的で、ボトルネックは CUDA0/3 の不均衡（H3）。expert を選択的に GPU へ戻す場合、空きが大きい CUDA1/2 が担当する中間層を狙うのが有効。

## 採用構成の起動コマンド（再現手順）

```bash
# 1. ロック取得
.claude/skills/gpu-server/scripts/lock.sh t120h-p100

# 2. 既存プロセス停止（稼働中のみ）
.claude/skills/llama-server/scripts/stop.sh t120h-p100

# 3. 起動（ssh 経由で直接投入）
MODEL_PATH="/home/llm/.cache/huggingface/hub/models--unsloth--Qwen3.5-122B-A10B-GGUF/snapshots/51eab4d59d53f573fb9206cb3ce613f1d0aa392b/Q4_K_M/Qwen3.5-122B-A10B-Q4_K_M-00001-of-00003.gguf"

LAUNCH_CMD="./build/bin/llama-server \
  -m '$MODEL_PATH' --jinja \
  -ngl 999 -ot 'blk\\.([0-9]|1[0-3]|2[0-9]|3[0-9]|4[0-7])\\.ffn_.*_exps\\.weight=CPU' \
  --flash-attn 1 --poll 0 -b 8192 -ub 8192 \
  --n-predict 32768 --threads -1 \
  --ctx-size 131072 --parallel 1 \
  --cache-type-k q8_0 --cache-type-v q8_0 \
  --defrag-thold 0.1 \
  --temp 0.6 --top-p 0.95 --top-k 20 --min-p 0 \
  --port 8000 --host 0.0.0.0 \
  --alias 'unsloth/Qwen3.5-122B-A10B-GGUF:Q4_K_M'"

ssh -f t120h-p100 "cd ~/llama.cpp && nohup bash -c \"\$LAUNCH_CMD\" > /tmp/llama-server.log 2>&1 < /dev/null &"

# 4. ヘルスチェック
until curl -sf http://10.1.4.14:8000/health > /dev/null; do sleep 5; done

# 5. eval ベンチ
curl -s http://10.1.4.14:8000/v1/chat/completions \
  -H 'Content-Type: application/json' \
  -d '{"model":"unsloth/Qwen3.5-122B-A10B-GGUF:Q4_K_M","messages":[{"role":"user","content":"..."}],"max_tokens":256}'

# 6. ロック解放
.claude/skills/gpu-server/scripts/unlock.sh t120h-p100
```
