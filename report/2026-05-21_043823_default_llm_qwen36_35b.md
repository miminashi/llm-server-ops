# デフォルトLLMをQwen3.6-35B-A3Bに切り替え

- **実施日時**: 2026年5月21日 04:38 JST
- **作業者**: Claude
- **作業種別**: 設定変更（`llama-up.sh` デフォルト引数 + ドキュメント追従）

## 核心発見サマリ

opencode 実ワークロードベンチ（2026-05-20 〜 21）で `unsloth/Qwen3.6-35B-A3B-GGUF:UD-Q4_K_XL` が以下を満たし、現行運用モデル `unsloth/Qwen3.5-122B-A10B-GGUF:Q4_K_M` の置換として確定した。

- judge_score **4.44**（全モデル中最高、122B 比 +0.55）
- wall_time **365s vs 122B 1047s**（×2.9 高速、eval_tps 12.0）
- 9/9 全完走（唯一の 100% 成功）
- MTP 版は tool-heavy ワークロードで draft 無効化により逆効果、判定式失格

本作業では `llm-server-ops` のデフォルトLLM（`llama-up.sh` 引数省略時の起動モデル）を Qwen3.6-35B-A3B-GGUF:UD-Q4_K_XL に切り替えた。

## 前提・目的

- **背景**: opencode のベンチで Qwen3.6-35B-A3B が q35-122b を判定式上回り、運用切替が推奨された。`llm-server-ops` 側のデフォルトもこれに追従し、引数なしの `llama-up.sh` 起動で推奨モデルが立ち上がるようにする。
- **目的**: `llama-up.sh` 引数省略時のデフォルトを Qwen3.6-35B-A3B-GGUF:UD-Q4_K_XL / ctx=131072 に変更し、関連ドキュメントを統一する。
- **前提条件**:
  - llama.cpp HEAD（2026-05-16 以降の MTP サポート版）が GPU サーバ側にビルド済み
  - HuggingFace キャッシュに `unsloth/Qwen3.6-35B-A3B-GGUF:UD-Q4_K_XL` が存在（初回未取得時は llama-server 起動時に自動 pull）

## 環境情報

- **GPU サーバ**: t120h-p100（NVIDIA Tesla P100-PCIE-16GB × 4、VRAM 64GB、10.1.4.14）
- **想定動作モード**: 通常起動（CPU offload なし）、ctx=131072
- **サンプリングパラメータ**: `--temp 0.6 --top-p 0.95 --top-k 20 --min-p 0`（`start.sh` 既存の Qwen3.6 分岐で自動適用）
- **MTP**: 非適用（デフォルトモデルは非MTP）

**MODE 変更理由**: Qwen3.6-35B-A3B には `start.sh` 側に fit プロファイルが定義されていない。`MODE=fit` のままだと汎用 fit ロジック（`ffn_.*_exps.weight=CPU` 全層オフロード + ctx=8192）が走り、ベンチ結果（ctx=131072 通常起動で eval 12 t/s）を再現できないため、デフォルトを `131072` に変更した。旧 122B を使う場合は第3引数で明示的に `fit` を渡す。

## 影響範囲

- **引数なしで `llama-up.sh` を実行している既存ワークフロー**: 起動モデルが Qwen3.5-122B-A10B から Qwen3.6-35B-A3B に切り替わる。eval 速度向上、判定品質向上のため基本的に利点のみ。
- **CI/自動化スクリプトなど明示的にモデル名を渡している箇所**: 変更なし。
- **API クライアント側のモデル名指定**: `model` フィールドに旧モデル名 (`Qwen3.5-122B-A10B-GGUF:Q4_K_M` 等) をハードコードしているクライアントは、新モデルを起動した状態だと llama-server が `model not found` 相当の応答を返す可能性あり。クライアント側で `/v1/models` から動的取得するか、明示的に新モデル名へ更新する必要がある。
