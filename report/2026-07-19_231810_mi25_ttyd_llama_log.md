# mi25 で ttyd 経由 llama-server ログ閲覧を開通

- **実施日時**: 2026年7月19日 22:50 〜 23:15 JST (skill/コード実態調査、mi25 での ttyd バイナリ有無確認、`ttyd-up.sh` 単体起動テスト、SKILL.md 補足)
- **報告日時**: 2026年7月19日 23:18 JST
- **作成者**: Claude Opus 4.7 (1M context)

## 概要

mi25 で llama-server を起動しても、他 2 サーバ (t120h-p100 / t120h-m10) で使えていた ttyd 経由のブラウザからのログ閲覧が使えないという要望を受けて対応した。

調査の結果、llama-server スキル側のコードは 2026-06-10 の「ttyd 起動信頼性向上」対応で既に mi25 分岐まで実装済みで、`start.sh mi25 ...` を叩けば ttyd 起動が自動で呼ばれる設計になっていた。未対応の実体は 1 点だけで、mi25 側に `ttyd` バイナリが導入されていなかったため、起動処理が silent fail し続けていた。

本対応ではまずユーザ側で `sudo apt install ttyd` を実施してもらい、その後、既存の `ttyd-up.sh` を単体で mi25 に対して起動して動作を確認した。llama-server は他タスクで使用中だったため、`start.sh` の停止・再起動を伴う統合確認は今回スコープ外とし、次回の自然な再起動機会に確認することとした。

結果として、当初の要望である「llama-server ログを ttyd で閲覧できる状態にする」は 7682 番ポートで達成できた。稼働中の llama-server プロセスやサービスに一切影響を与えず、追加ポート 7682 を新たに開けるだけで完結している。

一方で副次的な発見として、GPU 監視用の 7681 番ポートについては、既に別プロセスの ttyd が loopback バインドで占有しており、`ttyd-up.sh` が試みる kill パターンにマッチせず、そのままそちらが居座り続ける現象が観測された。LISTEN 検証は「何かが LISTEN していれば OK」と判定するため、この状態は自動テストでは silent に見逃される。今回のユーザ要望はログ閲覧 (7682) なので直接影響はないが、`ttyd-up.sh` の pkill パターン設計の corner case として残課題に記録する。

再発防止として、SKILL.md の「ttyd-up.sh の動作」節に、サーバ側 `ttyd` バイナリの導入が暗黙前提であった点を 1 行追記した。次回同種のサーバで同じ躓きが起きないよう、導入確認コマンドも併記している。

## 前提・目的

- **背景**: mi25 で llama-server 起動時に他サーバと同じ ttyd 経由のブラウザ閲覧 (7681 GPU 監視 / 7682 ログ) を使えるようにしたい
- **目的**: mi25 実機での ttyd (少なくとも 7682 ログ閲覧) の疎通確立、および原因の SKILL.md 明文化
- **前提条件**: llama-server スキル (`ttyd-up.sh`, `start.sh`) は 2026-06-10 対応で既に mi25 分岐を含む。mi25 は本対応中他タスクで使用中のため、llama-server の停止・再起動は行わない

## 環境情報

- 対象サーバ: mi25 (10.1.4.13)
- OS: Ubuntu 22.04 (jammy)
- ttyd: 1.6.3 (`/usr/bin/ttyd`, `apt-cache policy` 経由の `1.6.3+20210924-1build1`。ユーザが本対応中に `sudo apt install ttyd` で導入)
- 対象スクリプト: `.claude/skills/llama-server/scripts/ttyd-up.sh`, `.claude/skills/llama-server/scripts/start.sh`
- 稼働中の llama-server: pid 3454450 (port 8000 で LISTEN)

## 再現方法

1. mi25 での ttyd バイナリ導入確認:
   ```bash
   ssh mi25 "command -v ttyd && ttyd --version 2>&1 | head -1"
   # 期待: /usr/bin/ttyd と ttyd version 1.6.3
   ```

2. `ttyd-up.sh` 単体起動 (llama-server 稼働中でも安全 = 7681/7682 のみ操作、ロック不要):
   ```bash
   .claude/skills/llama-server/scripts/ttyd-up.sh mi25
   ```

3. サーバ側 LISTEN・プロセス確認:
   ```bash
   ssh mi25 "ss -tlnp 2>/dev/null | grep -E ':(7681|7682) '; pgrep -a ttyd"
   ```

4. ブラウザで `http://10.1.4.13:7682` を開き、`tail -f /tmp/llama-server.log` が流れることを確認 (今回本文では未実施、次アクション)。

## 結果詳細

### `ttyd-up.sh mi25` の出力

```
==> ttyd を mi25 で起動中 (port 7681 GPU監視 / 7682 ログ閲覧)...
Pseudo-terminal will not be allocated because stdin is not a terminal.
    [OK] GPU監視  : http://10.1.4.13:7681
    [OK] ログ閲覧 : http://10.1.4.13:7682
==> ttyd 起動完了（7681/7682 とも LISTEN）
```

スクリプト自身は両ポート成功で return したが、サーバ側の実プロセスと LISTEN 状態を照合すると 2 つのポートで実態が異なっていた。

### サーバ側の実態 (`ss -tlnp` + `pgrep -a ttyd`)

```
LISTEN 0 512      0.0.0.0:8000      users:(("llama-server",pid=3454450,fd=13))   ← 稼働中 llama-server (無関係)
LISTEN 0 4096     0.0.0.0:7682      users:(("ttyd",pid=3508471,fd=11))           ← 今回起動 (成功)
LISTEN 0 4096   127.0.0.1:7681                                                    ← 既存の別 ttyd (loopback のみ)

3492386 /usr/bin/ttyd -i lo -p 7681 -O login                                      ← 既存 (今回のスクリプトが起動したものではない)
3508471 ttyd --port 7682 --writable bash -c tail -f /tmp/llama-server.log         ← 今回起動 (正しい形)
```

### ポート別の評価

| ポート | 今回起動の ttyd | 実 LISTEN | 外部到達性 | ユーザ要望に対して |
|--------|----------------|-----------|-----------|-------------------|
| 7682 (ログ閲覧) | pid 3508471, `--writable bash -c 'tail -f /tmp/llama-server.log'` | `0.0.0.0:7682` | 到達可 | **達成** |
| 7681 (GPU 監視) | 起動しようとしたが 7681 が既存 ttyd に取られていて bind 失敗 (LISTEN は既存の別 ttyd) | `127.0.0.1:7681` のみ (別 ttyd) | ブラウザから不可 | 副次 (今回スコープ外) |

## 副次発見

### `ttyd-up.sh` の pkill パターンが「フルパス指定で起動された別 ttyd」に効かない

`ttyd-up.sh:64` の停止処理:
```bash
ssh "$SERVER" "pkill -f '^ttyd --port 7681' 2>/dev/null; ...; true"
```

`^ttyd --port` で先頭アンカーしているため、コマンドラインが `/usr/bin/ttyd -i lo -p 7681 -O login` の形で `-p` を使っている、あるいはフルパスで始まる ttyd プロセスにはマッチしない。今回 mi25 で観測された既存 7681 の ttyd (pid 3492386) はまさにその形態で、`ttyd-up.sh` は kill を諦め、自分の版を 7681 に bind しようとして (7681 は既存に取られているため) 起動に失敗する。ところが LISTEN 検証 (`ss -tln | grep -q ':7681 '`) は既存 ttyd を誤検知して「OK」を返し、外部から見えない状態のまま silent success する。

2026-06-10 レポートの pkill 自己マッチ対応 (`^ttyd --port` アンカー化) は正しく機能しているが、「別ユーザ流儀で先に立った ttyd」への対処は残っていた形。

### 7681 側の既存 ttyd の素性は不明

`-O login` オプションは ttyd の TLS/認証設定に関わるフラグ (詳細は今回未追跡)。誰が何のために立てたかは不明。loopback バインドなので外部から見えていた形跡はない。触ると副作用の恐れがあるため今回は放置。

## 残課題

- **`start.sh` からの統合起動確認**: mi25 で llama-server を次回自然に再起動する機会に、`start.sh mi25 ...` の末尾で 7681/7682 の自動起動が期待通り出るか確認する。
- **ブラウザ end-to-end 確認**: `http://10.1.4.13:7682` で `tail -f /tmp/llama-server.log` が実際に描画・追記ストリームされることの目視確認。
- **7681 の既存 ttyd 問題**: mi25 上の pid 3492386 (`/usr/bin/ttyd -i lo -p 7681 -O login`) の素性確認と処遇決定。GPU 監視 UI が本当に必要になった時点で対応で十分。
- **`ttyd-up.sh` の pkill パターン強化**: `^ttyd --port` に加えて `-p ` オプション形態、フルパス起動形態にも効くパターンへの拡張。ただし副作用 (別サービス誤 kill) の吟味必要。優先度低。

## 結論・対応

- ユーザ要望「mi25 で ttyd 経由の llama-server ログ閲覧」は **7682 番ポートで達成済み** (`http://10.1.4.13:7682`)。
- SKILL.md に「サーバ側 `ttyd` バイナリの導入が前提である」旨を追記した。次回同種のサーバ (Ubuntu 系) では `sudo apt install ttyd` の 1 コマンドで開通する。
- 7681 (GPU 監視) は副次的問題として残課題に記録し、今回スコープ外。
