## 調査で分かったこと

### 7.1 モデル構造

- `n_layer=48`、`n_head_kv=2`、`n_embd_k_gqa=n_embd_v_gqa=512`、`n_expert=256`、`n_expert_used=8`、`n_ctx_train=262144`。
- hybrid 構造: 48 層のうち一部だけが attention、残りは recurrent state（SSM 系）。これは KV cache の実測が計算値より極端に小さいこと（後述）と、`llama_memory_recurrent: RS buffer` ログ行の存在から判断した。

### 7.2 GPU 別メモリ内訳（ctx 65535 時）

ログの `load_tensors` / `llama_kv_cache` / `llama_memory_recurrent` / `sched_reserve` から抽出:

| GPU | model buf | KV buf | RS buf | compute buf | ログ合計 | nvidia-smi |
|---|---|---|---|---|---|---|
| CUDA0 | 1301 MiB | 204 MiB | 41 MiB | **4576 MiB** | 6122 MiB | 7495 MiB |
| CUDA1 | 1199 MiB | 204 MiB | 37 MiB | 2337 MiB | 3776 MiB | 4691 MiB |
| CUDA2 | 1199 MiB | 204 MiB | 37 MiB | 2337 MiB | 3776 MiB | 4691 MiB |
| CUDA3 | 1693 MiB | 204 MiB | 33 MiB | **8048 MiB** | 9978 MiB | 10891 MiB |

- 各 GPU の空き（nvidia-smi）: CUDA0 = 8776 MiB / CUDA1 = 11580 MiB / CUDA2 = 11580 MiB / **CUDA3 = 5380 MiB**。
- 合計 VRAM 使用量: 約 27.5 GiB / 64 GiB。**空き約 37 GiB**。

### 7.3 CPU オフロードされた expert 重み

- ログより `CPU_Mapped model buffer size = 47000.12 MiB + 25271.75 MiB = 72271.87 MiB`
- **合計 72.27 GiB が CPU 側にオフロードされている**（これが `-ot 'ffn_.*_exps.weight=CPU'` で外された expert 重み）。
- 1 層あたり: 72.27 / 48 ≈ **1.506 GiB/層**。
- プロセス RSS は 82.9 GiB（mmap された GGUF ファイルの resident 分 + CPU 保持の expert）。RAM は十分に余裕（available 234 GiB）。

### 7.4 compute buffer が支配的

- compute buffer だけで合計 17.3 GiB（4576 + 2337 + 2337 + 8048）。model buffer の合計 5.4 GiB を大きく上回る。
- 特に **CUDA3 の 8048 MiB** が突出 → これが GPU3 を最も逼迫させている要因。
- CUDA0 の 4576 MiB は embedding 関連と推定。

### 7.5 KV buffer が小さい理由

- Q8_0 の理論値（単純 GQA 計算）:
  - K per token = `n_layer × n_embd_k_gqa × 1.0625 = 48 × 512 × 1.0625 ≈ 26112 B`
  - V per token も同じ → KV per token ≈ 51 KB
  - ctx 65536 → 51 KB × 65536 ≈ 3.3 GiB
- 実測: 204 MiB × 4 GPU = **816 MiB**（理論の約 1/4）。
- **解釈**: hybrid 構造のため、48 層のうち attention 層は 12 層程度しかない（残りは recurrent で KV 不要）と推定される。結果、ctx を倍にしたときの KV 追加も小さい。

### 7.6 ctx 倍化（65536 → 131072）の VRAM インパクト

- KV buffer: 204 → **408 MiB/GPU**（+204 MiB/GPU、合計 +816 MiB）
- RS buffer: 変化なし（recurrent state は ctx 非依存）
- model buffer: 変化なし
- compute buffer: **flash-attn 1 が有効なら理論的には ctx 非依存**（活性化は `-ub` 依存、QK^T は FA により非物質化）。ただし実測での検証が必要（後述の仮説 H2）。

→ **仮説 H1 が正しければ、ctx 131072 化で増える VRAM は各 GPU あたり +204 MiB のみ**。GPU3 でも空き 5176 MiB 残る見込み。

## 対立する仮説

| ID | 仮説 | 根拠 | 帰結 |
|---|---|---|---|
| **H1** | compute buffer は ctx-size にほぼ非依存 | flash-attention は O(N) メモリ、sched_reserve は `-ub` と model 構造が主体 | ctx 131072 化で +204 MiB/GPU のみ。プラン A で安全。 |
| **H2** | compute buffer は ctx-size に弱く比例 | 一部のスクラッチ領域（`ggml_cgraph` サイズ、SWA リングバッファ等）は n_ctx に依存することがある | GPU3 が 8 → 9〜10 GiB に膨張、空き 4 GiB 未満。プラン B の余地が消える。 |
| **H3** | GPU 間不均衡は `--split-mode layer` の既定計算起因で、`--tensor-split` 介入で緩和できる | model buffer が GPU0（embed）と GPU3（output）で大きい、compute buffer も偏る | 各 GPU に自由に重みを足すには `--tensor-split` の手動調整が必要。 |

これらは互いに排他ではなく **複数同時に成り立ち得る**。H1 と H3 が両立する場合、プラン A+B が有望。H2 が顕著な場合、compute buffer を先に縮めるプラン D が必要になる。

## 提案（複数プラン併記）

「ctx 128k を確保しつつ expert 重みを VRAM に戻す」という依頼に対し、仮説の不確実性を踏まえて 4 プランを示す。各プランは **狙い／差分コマンド／期待値／リスク** をセットで記載した。

### プラン A: ctx 131072 化のみ（最小変更・ベースライン）

- **狙い**: 仮説 H1 の実地検証。expert 配置は現状維持。最小の差分で 128k が本当に収まるかを確認する「測定ステップ」。
- **差分**: `--ctx-size 65535` → `--ctx-size 131072`（それ以外は現状維持）
- **推奨実行手段**: `start.sh` の fit モード引数が第 4 引数にそのまま渡るため、下記で出せる（現状と `-ot` パターンが同一）。

  ```bash
  .claude/skills/llama-server/scripts/start.sh t120h-p100 \
    "unsloth/Qwen3.5-122B-A10B-GGUF:Q4_K_M" fit 131072
  ```

  `start.sh:206-208` で `NGL_OPTS="-ngl 999 -ot 'ffn_.*_exps.weight=CPU'"`、`CTX_OPTS="--ctx-size $FIT_CTX"` となる。
- **期待 VRAM**:
  - CUDA0: 7495 → 約 7699 MiB
  - CUDA1: 4691 → 約 4895 MiB
  - CUDA2: 4691 → 約 4895 MiB
  - **CUDA3: 10891 → 約 11095 MiB（空き 5289 MiB）**
- **期待 eval 速度**: 変化なし（~10 t/s）。expert は依然 CPU。
- **リスク**: H2 が強く効く場合、CUDA3 で compute buffer が膨らんで OOM。
- **退避プラン A′**: OOM したら `-b 4096 -ub 4096` に縮小してリトライ。

### プラン B: down_proj のみ GPU に復帰（`ffn_(gate|up)_exps` を CPU）

- **狙い**: expert の 3 種テンソル（gate/up/down）のうち **down のみ GPU に戻す**。全 48 層均等に戻るので GPU 間不均衡が自然に回避される（仮説 H3 への介入不要）。
- **差分**:
  ```
  -ot 'ffn_(gate|up)_exps\.weight=CPU'
  --ctx-size 131072
  ```
- **追加 VRAM 見積もり**:
  - expert 全体 72.27 GiB のうち down のみ → 約 1/3 = **24.1 GiB を GPU に復帰**
  - 4 GPU 均等なら約 6 GiB/GPU 追加
  - CUDA3 の空き 5380 MiB → **ギリギリ足りない可能性あり**（6 GiB > 5.4 GiB）
- **期待 eval 速度**: down_proj 計算が GPU 側で行えるため、MoE 推論のクリティカルパスが一部 GPU 化。粗見積もり **10 → 13〜17 t/s**。
- **リスク**: CUDA3 OOM。GPU 間で layer split が不均衡（GPU3 が 12 層超なら特に）。
- **実装コスト**: `start.sh` の fit モード分岐の `-ot` 文字列（`start.sh:207`）を書き換える必要があるか、手動起動でコマンド直打ち。
- **バリエーション B′**: `ffn_(gate|up)_exps` が NG なら `ffn_(up|down)_exps\.weight=CPU`（gate のみ GPU）、`ffn_(gate|down)_exps\.weight=CPU`（up のみ GPU）などで最小 OOM な組を探す。

### プラン C: 層単位で選択的に復帰

- **狙い**: 仮説 H3 を前提に、空きの多い GPU1/GPU2 が担当すると見られる中間層だけ expert を GPU に戻す。
- **層 → GPU マッピング推定**（model buffer サイズ比から）:
  - CUDA0: layer 0〜11（embed 含む、1301 MiB）
  - CUDA1: layer 12〜23（1199 MiB）
  - CUDA2: layer 24〜35（1199 MiB）
  - CUDA3: layer 36〜47（output 含む、1693 MiB）
- **llama.cpp `-ot` の仕様**: tensor 名にマッチしたテンソルを CPU に送るか GPU に残すかの 2 値。「どの GPU に置くか」は指定できない。配置は `--split-mode layer` の自動計算に従う。
- **C-1 (控えめ)**: layer 14〜19（6 層、+9 GiB）を GPU に戻す。主に CUDA1 に集中するので CUDA3 の負荷は増えない。
  ```
  -ot 'blk\.([0-9]|1[0-3]|2[0-9]|3[0-9]|4[0-7])\.ffn_.*_exps\.weight=CPU'
  --ctx-size 131072
  ```
  - 期待: eval ~12 t/s、CUDA1 使用 約 13.7 GiB、リスク小。
- **C-2 (攻め)**: layer 12〜27（16 層、+24 GiB、CUDA1 と CUDA2 に分散）
  ```
  -ot 'blk\.([0-9]|1[01]|2[89]|3[0-9]|4[0-7])\.ffn_.*_exps\.weight=CPU'
  --ctx-size 131072
  ```
  - CUDA1: +8 層 = +12 GiB → 合計 約 16.7 GiB → **16 GB 超で不可**
  - 層数を更に絞る（例 layer 12〜23 のみで +18 GiB、CUDA1 に集中）必要あり。要実測。
- **リスク**: layer → GPU マッピングの推定が外れると、想定外の GPU が OOM する。実地で `load_tensors` ログを再確認する必要あり。

### プラン D: `-b/-ub` 縮小で compute buffer を削り、B を成立させる

- **狙い**: プラン B の最大の障壁は CUDA3 の 8 GiB compute buffer。`-b 4096 -ub 4096` にすれば CUDA3 の compute buffer は概ね半分（4 GiB 前後）まで落ちると期待でき、その分 expert を多く戻せる。
- **差分**:
  ```
  -b 4096 -ub 4096
  -ot 'ffn_(gate|up)_exps\.weight=CPU'
  --ctx-size 131072
  ```
- **トレードオフ**:
  - ✗ prompt processing が遅くなる（pp512: 693 → 400 t/s 相当の予想、現 prompt eval ~140 t/s → ~80 t/s へ）
  - ✓ CUDA3 空きが +4 GiB → プラン B が収まる見込みが立つ
  - eval（生成）速度には `-ub` 低下の直接影響は小さい（B の復帰効果がそのまま出る）
- **期待 eval 速度**: B と同等 or やや上（安定して 13〜17 t/s）。
- **用途**: B 単独で OOM した場合の保険策。

### 推奨実行順（段階検証）

1. **プラン A** を 1 回だけ起動し、ctx 131072 で GPU3 に十分余裕があるか確認。仮説 H1 vs H2 を実測で判定。
2. A が安定かつ GPU3 に 5 GiB 以上空きがあれば → **プラン B** を試行し eval 速度を再計測。
3. B が OOM したら → **プラン D**（`-b 4096 -ub 4096` 併用）で再試行。
4. D でも収まらない場合 → **プラン C-1**（layer 14〜19 のみ）にフォールバック。
5. 余裕があれば C の適用層を順次拡張して最大化。

## 参考にした計算式

### KV cache per token（Q8_0）

```
K per token = n_layer_attn × n_head_kv × n_embd_head_k × bytes_per_elem
V per token = n_layer_attn × n_head_kv × n_embd_head_v × bytes_per_elem
bytes_per_elem (Q8_0) ≈ 1.0625 B（1 byte の quant + block scale のオーバーヘッド）
```

実測（ctx 65536、合計 816 MiB）から逆算すると:
```
816 MiB / 65536 tokens ≈ 12.75 KB/token
→ n_layer_attn × n_head_kv × (256+256) × 1.0625 ≈ 12750
→ n_layer_attn ≈ 12 層相当（残り 36 層は recurrent と推定）
```

### 1 層あたり expert サイズ

```
CPU_Mapped total ≈ 72271 MiB
/ 48 layers ≈ 1506 MiB/layer = 1.506 GiB/layer
```

これが「層単位で expert を GPU に戻したとき」の VRAM 追加量の目安。
