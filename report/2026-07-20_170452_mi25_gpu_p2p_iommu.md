# mi25 4 枚 P2P — 同 NUMA 9.6 GB/s、越境 0.24 GB/s、IOMMU 無関与

- **実施日時**: 2026年7月20日 16:22 〜 17:05 JST (Phase 1 静的環境確認 → hip_peer_check before → GRUB 更新 + 再起動 → BIOS 状態判明 → hip_peer_check + llama-bench after)
- **報告日時**: 2026年7月20日 17:05 JST
- **作成者**: Claude Opus 4.7 (1M context)

## 概要

mi25 サーバは MI25 (Vega10 gfx900) 4 枚を積んで実運用しているが、GPU 間の「ピア間通信 (P2P)」が実効的に効いているのか、それとも内部的にホストメモリ経由でコピーしてしまっているのかが今まで測られていなかった。過去のレポートには `rocm-smi --showtopo` のトポロジ観察 (Link Type=PCIE、XGMI 無し) しかなく、hipMemcpyPeer や rocm-bandwidth-test での実測は皆無だった。今回、実際に GPU から GPU へメモリコピーを行うマイクロベンチを書いて帯域を測り、IOMMU の状態を含めた環境要因も併せて評価した。

まず現状のカーネル引数と dmesg を確認した段階では、`iommu=` 系の引数もメッセージも見えなかったため「IOMMU は無効」と推定していた。しかし実測に入ってみると、同一 NUMA 内 (SLOT2-4 と SLOT6-8 のそれぞれ 2 枚ずつ) の GPU 間コピーは約 9.6 GB/s と PCIe 3.0 x16 の理論値の 60% 相当が出て健全であるのに対し、CPU の NUMA 境界を跨ぐ組み合わせは 0.24 GB/s と 40 倍低い値に落ちるという極端な非対称が観測された。これは llama.cpp の `--split-mode row` 相当のワークロードでは致命的になる。

原因を切り分けるため、ユーザから許可を得た上で GRUB に `iommu=pt intel_iommu=on` を追加し再起動して after 測定を実施した。ここで初めて `sudo dmesg` を確認したところ、実は BIOS 側の VT-d は最初から Enabled になっており DMAR ACPI テーブルが有効だったこと、また before 段階では kernel 側が IOMMU を使っていなかっただけであったことが判明した (一般ユーザ権限では kernel_dmesg_restrict でメッセージが見えなかったため誤診していた)。

肝心の after の帯域は before と実質同一で、同一 NUMA 内は 9.5-9.6 GB/s、cross-NUMA は 0.24-0.25 GB/s と誤差以内の値だった。llama-bench (Qwen3-VL-30B-A3B と gpt-oss-20B の split-mode layer) でも pp512 と tg128 に有意差は見られなかった。つまり IOMMU を isolation モードにしたり passthrough モードにしたりしても、この構成の実効帯域には影響しない。

結論として、mi25 の cross-NUMA での P2P 帯域が低いのは IOMMU の設定問題ではなく、この世代の Intel Xeon E5-2600 v3/v4 (Haswell/Broadwell EP) 環境で QPI (CPU 間相互接続) を跨ぐ PCIe P2P が実効的に成立せず、ROCm/HIP ランタイムが host bounce (RAM 経由の 2 段コピー) にフォールバックしているためと推定される。ソフトウェア側 (IOMMU/kernel パラメータ) で回避できる余地はなく、大量のカード間転送を必要とするワークロードでは同一 NUMA 内 (最大 2 枚) で完結させるか、あるいはそもそも cross-NUMA 転送が発生しない split 戦略 (`--split-mode layer` はレイヤー境界のみで少ない) を選ぶしかない。

副次的に、llama.cpp の Vulkan バックエンドは `--split-mode row` でモデルロードに失敗する (Qwen3-VL-30B-A3B と gpt-oss-20B の両方で確認) ことも分かった。Vulkan の tensor split は事実上 layer split のみで、cross-NUMA の 0.24 GB/s は実運用で顕在化しにくいという点で mi25 は運良く救われている。

今後、大きなカード間帯域が必要なワークロードを走らせたい場合は本サーバ (mi25 = Supermicro X10DRG-Q + Xeon E5-2600 v3/v4) では避け、より新しい世代 (Skylake-SP 以降で単一 IIO 化 or CXL / NVLink 対応機) を検討する材料になる。

## 核心発見サマリ

![mi25 4 枚 GPU 間 hipMemcpyPeer 帯域 (before/after IOMMU passthrough)](attachment/2026-07-20_170452_mi25_gpu_p2p_iommu/summary.png)

**結論**: mi25 の GPU 間 P2P は**同一 NUMA 内 (D0-D1 / D2-D3) で 9.5-9.6 GB/s** (PCIe 3.0 x16 の理論 15.75 GB/s の約 60%、以前から実効的に効いていた) だが、**cross-NUMA (D0/1 ↔ D2/3) は 0.24-0.25 GB/s** と 40 倍低い。**原因は Intel Xeon E5-2600 v3/v4 (Haswell/Broadwell EP) 環境で QPI 越えの PCIe P2P が実効的に成立せず、ROCm ランタイムが host bounce (RAM 経由) にフォールバックしているためと推定** (実測観察に基づく、Intel 仕様書での "non-support" は未直接確認)。BIOS の Intel VT-d は最初から Enabled、kernel に `iommu=pt intel_iommu=on` を追加して passthrough モードに切替えた後 (before=isolation 未使用、after=passthrough active) でも帯域・llama-bench (pp512/tg128) に有意差なし。ソフト側改善の余地なし。

## 再現方法

### Phase 1: 環境確認 (read-only)

```bash
ssh mi25 "sudo lspci -vvv -s 04:00.0 | grep -E 'Region 0'"   # Large-BAR (16G)
ssh mi25 "cat /sys/module/amdgpu/parameters/pcie_p2p"        # Y
ssh mi25 "sudo dmesg | grep -iE 'IOMMU|DMAR|VT-d' | head"    # DMAR/passthrough を証跡化
```

### Phase 2: hip_peer_check.cpp を作成・ビルド・実行

`hip_peer_check.cpp` を mi25 の `~/` に置いて:

```bash
ssh mi25 "cd ~ && hipcc -O2 -o hip_peer_check hip_peer_check.cpp"
ssh mi25 "HIP_VISIBLE_DEVICES=0,1,2,3 ~/hip_peer_check 256"
```

このコードは 4x4 の `hipDeviceCanAccessPeer` を照会し、全ペアで `hipDeviceEnablePeerAccess` を試み、
256MB バッファを `hipMemcpyPeerAsync` で 10 回転送して帯域 (GB/s) を出す。

### Phase 4: IOMMU passthrough を有効化

```bash
# /etc/default/grub をバックアップ
ssh mi25 "sudo cp /etc/default/grub /etc/default/grub.bak_p2p_$(date +%Y%m%d_%H%M%S)"
# GRUB_CMDLINE_LINUX_DEFAULT に iommu=pt intel_iommu=on を追加
ssh mi25 'sudo sed -i "s|^GRUB_CMDLINE_LINUX_DEFAULT=\"\"|GRUB_CMDLINE_LINUX_DEFAULT=\"iommu=pt intel_iommu=on\"|" /etc/default/grub'
ssh mi25 "sudo update-grub"
# BMC で soft shutdown → on
.claude/skills/gpu-server/scripts/bmc-power.sh mi25 soft
.claude/skills/gpu-server/scripts/bmc-power.sh mi25 on
# 起動後 dmesg で passthrough 確認
ssh mi25 "sudo dmesg | grep -iE 'IOMMU|passthrough'"
```

### Phase 5: llama-bench (Vulkan)

```bash
ssh mi25 "GGML_VK_VISIBLE_DEVICES=0,1,2,3 ~/llama.cpp/build-vulkan/bin/llama-bench \
  -m ~/.cache/llama.cpp/unsloth_Qwen3-VL-30B-A3B-Instruct-GGUF_Qwen3-VL-30B-A3B-Instruct-Q8_0.gguf \
  -sm layer -ngl 999 -ctk q8_0 -ctv q8_0 -fa on -r 3"
```

## 結果詳細

### rocm-smi --showtopo の weight/hops と実測の整合 (重要)

```
================================ Weight between two GPUs =================================
       GPU0         GPU1         GPU2         GPU3         
GPU0   0            40           61           61           
GPU1   40           0            61           61           
GPU2   61           61           0            40           
GPU3   61           61           40           0            

================================= Hops between two GPUs ==================================
GPU0   0            2            3            3            
GPU1   2            0            3            3            
GPU2   3            3            0            2            
GPU3   3            3            2            0            
```

- **Weight 40 (hops=2)** = 実測 **9.5-9.6 GB/s** (P2P direct、GPU → downstream bridge → GPU)
- **Weight 61 (hops=3)** = 実測 **0.24-0.25 GB/s** (host bounce、GPU → downstream bridge → root complex → QPI → root complex → downstream bridge → GPU)

**運用上の重要な意味**: rocm-smi の hops=3 (異 root complex) を見た時点で「実効帯域が桁違いに落ちる」ことを予測できる。次回別サーバで多枚数 GPU 構成を組む時、hip_peer_check を回す前に `rocm-smi --showtopo` の hops 値だけで cross-NUMA ボトルネックの有無を判定できる。

### hipMemcpyPeer 帯域 (256MB buffer, 10 iter 平均)

**before (kernel iommu 未設定、IOMMU inactive):**

```
src\dst      D0      D1      D2      D3     (GB/s)
  D0         -    9.58    0.24    0.24
  D1      9.51       -    0.24    0.24
  D2      0.25    0.25       -    9.58
  D3      0.25    0.25    9.58       -
```

**after (kernel `iommu=pt intel_iommu=on`, passthrough active):**

```
src\dst      D0      D1      D2      D3     (GB/s)
  D0         -    9.58    0.25    0.25
  D1      9.51       -    0.25    0.25
  D2      0.24    0.24       -    9.59
  D3      0.24    0.24    9.59       -
```

差分は測定誤差以内 (± 0.01 GB/s)。全ペアで `canAccessPeer=1`、`hipDeviceEnablePeerAccess` 全 12 通り成功 (before/after とも)。

**帯域の解釈**:
- 同一 NUMA 内 (D0-D1 の SLOT2-4 と D2-D3 の SLOT6-8): 9.5-9.6 GB/s は PCIe 3.0 x16 (理論 15.75 GB/s) の約 60%、typical な PCIe P2P direct 実効値。
- cross-NUMA (D0/1 ↔ D2/3): 0.24 GB/s は host bounce (RAM 経由コピー) と推定される。DDR4 メモリ帯域 (20+ GB/s) や QPI (実効 10+ GB/s) の生の帯域律速ではないため、ROCm/HIP ランタイム側で cross-CPU-socket 検出時に bounce buffer 経由の同期的な 2 段コピー (GPU_src → RAM → GPU_dst) に fallback している実装が支配的と考えられる。**根本原因の断定には ROCm ランタイムのソース確認が必要**。

### llama-bench (Vulkan, split-mode layer)

| モデル | test | before | after | 差分 |
|---|---|---:|---:|---:|
| Qwen3-VL-30B-A3B Q8_0 | pp512 | 466.06 ± 12.02 | 457.31 ± 14.18 | -1.9% (誤差以内) |
| Qwen3-VL-30B-A3B Q8_0 | tg128 | 49.28 ± 0.20 | 49.28 ± 0.17 | 0% |
| gpt-oss-20B Q8_0 | pp512 | 589.68 ± 7.15 | 553.94 ± 10.67 | -6.0% (誤差近い) |
| gpt-oss-20B Q8_0 | tg128 | 58.62 ± 0.25 | 59.28 ± 0.39 | +1.1% |

IOMMU の状態 (kernel 側 inactive → passthrough active) は Vulkan バックエンドの実効性能に有意な影響を与えない。

### PCIe topology & ACS

- MI25 は各カードに 2 段の PCIe bridge (upstream 1022:1470 + downstream 1022:1471) + GPU 本体 (1002:6860) の 3 段構造
- 上流 bridge (`02:00.0` 等) は ACS Extended Capability 無し (Vega10 内蔵)
- 下流 bridge (`03:00.0` 等) は ACSCap+ 全機能あり、しかし **ACSCtl は全 disabled** → P2P transaction がそのまま通る (P2P に有利)
- root complex 側 (Intel `[0000:80]-02.0`, `03.0`) は同一 root port から MI25 の SLOT6/SLOT8 が並列に生えている

### BIOS VT-d と kernel IOMMU の関係 (今回の学び)

| 状態 | kernel cmdline | 実効 | dmesg (要 sudo) |
|------|---------------|-----|-----------------|
| before | `ro` のみ | IOMMU inactive (kernel が使わない) | DMAR table 存在するが `IOMMU enabled` メッセージ無し |
| after | `iommu=pt intel_iommu=on` | Passthrough active | `DMAR: IOMMU enabled` / `Default domain type: Passthrough` |

**重要な学び**: 一般ユーザの `dmesg` は Ubuntu 22.04 の `dmesg_restrict=1` で kernel メッセージが見えないため、**必ず `sudo dmesg` で確認する**。今回は before 段階で「BIOS VT-d Disabled」と誤診していたが、実は最初から Enabled だった。

## 副次発見

### 1. canAccessPeer=1 と実効帯域は別物 (重要な学び)

hip_peer_check の canAccessPeer マトリクスは **全 4x4 で 1** (cross-NUMA も含む)、`hipDeviceEnablePeerAccess` も 12 通り全て成功 (before/after 両方) する。しかし cross-NUMA の実効帯域は 0.24 GB/s と host bounce レベル。つまり **HIP API 的には「P2P 可能」と応答するが、実際の転送は透過的に host bounce にフォールバックしている**。

これは重要な学びで、「canAccessPeer が真なら P2P が効いている」という直感は誤り。実効性能の判定には必ず **実測** が必要。判定基準としては:
- PCIe 3.0 x16 の 60% 以上 (≒9 GB/s 以上) → P2P direct
- 1 GB/s 未満 → host bounce (canAccessPeer が真でも実効的には P2P 無効)

なお同一 NUMA 内 (D0-D1 = SLOT2-4、D2-D3 = SLOT6-8) の 2 ペアは 9.51-9.59 GB/s と対称な値で、GPU 個体差 (c3164/448c4/c48c4/a48e4) や電力状態による差は観測されなかった。

### 2. Vulkan バックエンドは `--split-mode row` 未対応

Qwen3-VL-30B-A3B (MoE) と gpt-oss-20B (MoE) の両方で `-sm row` でモデルロード失敗 (`llama_bench: error: failed to load model`)。llama.cpp の Vulkan バックエンドは tensor split の row split を実装していない模様。実運用でも常に `--split-mode layer` (default) で動くため、cross-NUMA の 0.24 GB/s ボトルネックは顕在化しにくい (layer 境界の転送のみ、Q8_0 30B/20B での実測差 <7%)。

### 3. BMC (10.1.4.7) の RTT が高い

ローカル WS から BMC への ping RTT が 300-700ms と非常に高い。過去にはもっと低かった記憶があるので、経路上のネットワーク遅延か BMC 側の負荷。BIOS 進入試行で 1 回目 `bmc-power.sh mi25 status` が RMCP+ タイムアウトで失敗した (再試行で成功)。頻繁に BMC 操作するときは注意。

### 4. GRUB 変更後の POST → OS 起動が速く BIOS 進入に間に合わない

`bmc-power.sh mi25 on` から `bmc-kvm.py sendkeys Delete` を送るタイミングが 20 秒後だと、既に cloud-init の init-local ステージに突入していた。Supermicro X10DRG-Q は Above 4G Decoding の有無で POST 所要時間が大きく変わるため、次回 BIOS 進入試行時は電源 on 直後から Delete を数十回連打しつつスクショで進入を確認する運用が安全。今回は BIOS 変更が不要と判明したので進入は諦めた。

### 5. mi25 の実運用に影響する結論

同一 NUMA 内では 9.6 GB/s の P2P direct が既に効いているため、`--tensor-split 25,25,25,25` の layer split は問題なく動く。仮に llama.cpp Vulkan が将来 row split をサポートした場合、cross-NUMA 転送を含む split パターンは 40 倍低い帯域に律速されるため、`--tensor-split 50,50,0,0` (SLOT2/4 のみ) 等の同一 NUMA 内 2 枚に絞る運用の方が高速になる可能性がある。

## 残課題

- **rocm-bandwidth-test 未実施**: apt 経由で `sudo apt install rocm-bandwidth-test` すれば入手可能。今回は hip_peer_check で cross-NUMA と intra-NUMA の 40 倍差が明確に出たため必須ではないが、H2D/D2H/D2D の全パターンの補強実測として次回実施の余地あり。
- **NUMA affinity (numactl) 検証未実施**: 計画では `numactl --cpunodebind` で CPU node を GPU NUMA と揃える 2 水準比較を予定していたが未実施。cross-NUMA 帯域は QPI 非対応で救えないため意義は薄いが、host bounce 経路の性能を numactl で最適化できる可能性 (memory バインドで局所化) は残る。
- **1GB バッファでの hip_peer_check 未完了**: 計画では 256MB と 1GB の 2 サイズで測定予定だったが、1GB では cross-NUMA (0.24 GB/s × 12 ペア × 10 iter で数分/ペア) が長すぎて中断。256MB で 10 iter の平均が誤差 ±0.01 GB/s と十分安定していたため 1GB は不要と判断。
- **BIOS VT-d Disable 状態の実測**: 今回 BIOS 側は変更していない (最初から Enabled のまま)。isolation モード vs passthrough vs BIOS disable の 3 状態比較は完全ではない。ただしユーザ指示 (有効化して効果確認) は満たしており、実効性能に差がないと判明した今、追加検証の必要性は薄い。
- **llama.cpp Vulkan の row split サポート状況**: mainline に PR がないか、gate になっている理由 (RADV の shared memory 制約?) は要調査。将来サポートされたときに cross-NUMA を避ける `--tensor-split` を skill に追加する必要が出る。
- **kernel cmdline の常設化判断**: 現状 `iommu=pt intel_iommu=on` は入れっぱなしにしてある。実効性能に差がないので実害はないが、次回何か問題が出たら差し戻しの候補になる。`/etc/default/grub.bak_p2p_20260720_165208` からロールバック可。
- **llama-server の再起動**: Phase 4 で Qwen3.6-35B-A3B の llama-server (前セッションが起動していたもの) を stop.sh で停止したまま。必要なら `.claude/skills/llama-server/scripts/llama-up.sh mi25` で再起動可。

## 結論・対応

1. **mi25 4 枚 P2P は同 NUMA 内で既に実効的**: 9.6 GB/s (PCIe 3.0 x16 理論の 60%)。追加設定不要。
2. **cross-NUMA は 0.24 GB/s の壁**: Intel Xeon E5-2600 v3/v4 の QPI 越え PCIe P2P が実効的に非対応 (host bounce fallback、ROCm ランタイム実装含む要因)。BIOS/kernel の IOMMU 状態変更では改善不可 (今回実証)。
3. **IOMMU passthrough は無害**: kernel cmdline `iommu=pt intel_iommu=on` は残しても実効性能に影響なし。
4. **Vulkan は現状 `--split-mode layer` のみ**: cross-NUMA ボトルネックは顕在化しにくい。実運用の追加変更は不要。
5. **今後の大帯域必要ワークロード**: 本サーバ (Xeon E5-2600 世代) では避け、より新しい世代の検討材料とする。
