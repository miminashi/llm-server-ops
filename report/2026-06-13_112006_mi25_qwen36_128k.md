# mi25 で Qwen3.6-35B-A3B を 128k 実行（4枚復旧 + ub 最適化）

- **実施日時**: 2026年06月13日 11:20 (JST)

## 添付ファイル

- [実装プラン](attachment/2026-06-13_112006_mi25_qwen36_128k/plan.md)

## 核心発見サマリ

![ub スイープ: GPU[0] VRAM と速度](attachment/2026-06-13_112006_mi25_qwen36_128k/ub_sweep.png)

mi25 (MI25 ×4, gfx900/ROCm) で Qwen3.6-35B-A3B (UD-Q4_K_XL) を **131072 (128k) コンテキストで安定動作**させる構成を確定した。探索の過程で3つの想定外の壁を解決した。

1. **mi25 は実機 4 枚だが OS には 3 枚しか見えていなかった** — 原因は BIOS の **MMIO High Size 256GB 不足**。MI25 は 1 枚あたり 16GB の VRAM 全体を 64bit Large-BAR で公開するため、4 枚分の BAR 割当に MMIO 空間が足りず、BIOS が 4 枚目の PCIe Root Port (`80:02.0`) ごと無効化していた。**MMIO High Size を 512GB に拡大して 4 枚 64GB を復旧**。

2. **llama.cpp の最新 master は gfx900 でビルド不能** — コミット `112c78159` (NVFP4 dp4a kernel) 以降、`hip.h` が `__hip_fp8_e4m3` 型をアーキガードなしで参照し、gfx900 の device compile で `unknown type name` エラー。**ビルド可能な直前コミット `0fac87b15` (version 8533) に pin** して解決（`update_and_build-mi25.sh` を恒久修正）。

3. **gfx900 では ubatch を上げると遅くなる（直感に反する）** — GPU[0] への層偏り＋FA tile カーネルの compute buffer (ubatch 比例) で、`-ub 4096` はロード時点で GPU[0] 16.82GB（容量 17.16GB の 98%）に達し 32k プロンプトで **OOM クラッシュ**。`-ub 3072` は起動するが prompt **99.3 t/s** と遅く VRAM も 15GB と苦しい。**`-ub 2048` が prompt 122.8 t/s・GPU[0] 12.3GB と速度・VRAM 両面で最適**。

**確定構成**: `--flash-attn 1 --poll 0 -b 2048 -ub 2048`（KV q8_0、ctx 131072）。
131k で **OOM なし安定**、GPU[0] VRAM ピーク **12.3GB**（処理長に依らず一定、16GB に対し余裕 3.7GB）、**prompt 121.4 t/s / eval 23.5 t/s**（32k 計測、本番構成での再計測値。ub 探索時の単体計測 122.8 / 24.5 t/s と同等）。

## 前提・目的

- **背景**: P100 (t120h-p100) では Qwen3.6-35B-A3B を 128k で本番運用済み。2 台目の GPU サーバ mi25 でも同モデルを 128k で動かし、運用を冗長化したい。
- **目的**: P100 同様の探索を mi25 で行い、128k 実行の最適パラメータを確定。必要なインフラ整備（モデル配置・ビルド・start.sh 分岐）を行う。
- **前提条件**: mi25 が利用可能であること。モデルは P100 から LAN 経由でコピー（ユーザ指示）。

## 環境情報

- **サーバ**: mi25 (10.1.4.13)
- **GPU**: AMD Radeon Instinct MI25 ×4（gfx900 / Vega 10、VRAM 16GiB/枚、計 64GB）
- **プラットフォーム**: ROCm 6.2.2 (HIP 6.2.41134)、amdgpu 6.8.5
- **CPU/MB**: Xeon E5 v3 (Haswell-EP) ×2 ソケット、HPE 系（iLO 搭載）
- **llama.cpp**: `0fac87b15` (version 8533) — gfx900 ビルド可能コミットに pin（後述）
- **モデル**: `unsloth/Qwen3.6-35B-A3B-GGUF:UD-Q4_K_XL`（22.36GB、sha256 `707a55a8…f4450`、P100 からコピー）
- **比較対象**: P100 (NVIDIA Tesla P100 16GB ×4 = 64GB、CUDA sm_60)
- **ストレージ**: NVMe (nvme0n1p2 / ext4 ルート)、空き 132GB（モデル 22GB に十分）
- **sudo**: Phase 0 の診断（`sudo dmesg`・PCIe rescan）のため `llm` ユーザに NOPASSWD sudo を設定（`/etc/sudoers.d/llm-nopasswd`、ユーザ作業）

## Phase 0: 4 枚目 GPU 認識問題と解決

### 症状
`lspci | grep -c "Instinct MI25"` が **3**、llama-server も `found 3 ROCm devices`。ドキュメントの「4 枚 64GB」と相違。ユーザによれば物理的には 4 枚装着済み。

### 切り分け（PCIe トポロジ）
- GPU 用 PCIe Root Port: ソケット0 `00:02.0`/`00:03.0`(2枚) + ソケット1 `80:03.0`(1枚) = **3枚分のみ**。
- ソケット1 には対称な Root Port 2 (`80:02.0`) が dmesg・lspci 双方に**列挙されない**。

### 根本原因（dmesg）
```
pci 0000:84:00.0: BAR 0: no space for [mem size 0x400000000 64bit pref]   # 16GiB VRAM BAR
pci 0000:83:00.0: BAR 15: no space for [mem size 0x600000000 64bit pref]  # 24GiB bridge window
```
MI25 は 1 枚あたり 16GB VRAM 全体を 64bit prefetchable Large-BAR で公開する。BIOS の **MMIO High Size が 256GB** では 4 枚分の BAR＋ブリッジウィンドウ（アラインメント込み）を割り当てきれず、4 枚目の BAR 割当に失敗 → BIOS がその Root Port ごと無効化していた。`Above 4G Decoding` は既に Enabled だったが、MMIO High 容量が不足。

### 解決
BIOS で **MMIO High Size を 256GB → 512GB** に変更し再起動（MMIOHBase 3TB のため MMIO 領域は 3TB〜3.5TB、物理アドレス空間内）。結果:
```
lspci MI25 = 4 / rocm-smi GPU[0-3] 各16GB / llama-server: found 4 ROCm devices
```
**4 枚 64GB を復旧**。これにより P100 同等の VRAM 余裕を確保。

## Phase 1: モデルコピー + llama.cpp ビルド問題

### モデルコピー（P100 → mi25）
`transfer-file.sh`（Python http.server + curl、LAN 内転送）で GGUF を転送。**sha256 完全一致**で破損なしを確認（`707a55a8…f4450`、22,360,456,160 bytes）。
- 補足: `transfer-file.sh` は src に symlink を渡すとサイズ検証が symlink 自身の 76 bytes を拾い誤判定する小バグあり（転送機能自体は正常）。src には実体 blob パスを渡すのが確実。

### llama.cpp が gfx900 でビルド不能
`start.sh` の `update_and_build` が master を pull・ビルドする過程で失敗:
```
ggml/src/ggml-cuda/vendors/hip.h:252:9: error: unknown type name '__hip_fp8_e4m3'
1 error generated when compiling for gfx900.
```
- 原因: コミット `112c78159` "ggml-cuda: Add NVFP4 dp4a kernel (#20644)" 以降、`hip.h` が FP8 型 `__hip_fp8_e4m3` をアーキガードなしで参照。ROCm 6.2 では gfx900 の device compile でこの型が未定義となる。現 master (`c34b92235`, 本日付) でも未修正。
- P100/M10 は CUDA バックエンドのため影響なし（master 追従のまま）。
- なお `update_and_build-mi25.sh` の `build_llama_cpp()` は冒頭で `rm -rf build` するため、ビルド失敗で**既存の動作バイナリも消失**した。

### 解決（コミット pin）
FP8 型導入直前のコミット **`0fac87b15` (version 8533)** に checkout して再ビルド成功。`update_and_build-mi25.sh` を `git pull` から **`PINNED_COMMIT` への checkout** に変更し、master が gfx900 で再びビルド可能になるまで固定する設計に修正した。

## Phase 2: ub スイープと最適構成

`-fa auto` がデフォルトで Flash-Attention を有効化（gfx900 で動作、`flash_attn_ext_tile_case` カーネル使用）。FA workspace は ubatch に比例するため、ub をスイープした（131072 ctx / KV q8_0、32k プロンプトで計測。GPU[0] は処理ピーク）。

| ubatch | GPU[0] VRAM | prompt t/s | eval t/s | 結果 |
|--------|-------------|-----------|----------|------|
| 4096 | 16.82GB (ロード98%) | — | — | **32k で OOM クラッシュ** |
| **2048** | **12.33GB** | **122.8** | **24.5** | ✅ **最適** |
| 3072 | 14.95GB | 99.3 ↓ | 23.2 | 起動可だが遅く VRAM 苦しい |

- **ub を上げると prompt が遅くなる**（4096>3072>2048 の順で VRAM 増だが、速度は ub=2048 が最速）。gfx900 (Vega) では ub=2048 が演算効率の最適点で、それ以上は compute buffer 肥大でキャッシュ効率が落ちる。
- **VRAM ピークは処理長に依存しない**: KV は ctx=131072 分をロード時に全確保、FA workspace は ubatch 比例で一定。そのため 32k でも 131k 近傍でも GPU[0] は 12.3GB で一定。**131k 近傍（90k+ トークン）まで処理し OOM なし・VRAM 一定を実証**（速度計測の都合でフル 131k 完走前に打ち切り。詳細は「既知の課題」の prompt 速度を参照）。
- 速度・VRAM 両面で **ub=2048 が最適**と確定。tensor-split で GPU[0] 偏りを是正して ub を上げても、ub 自体が演算最適点を超えるため高速化しない。

### 本番構成での最終検証
`--flash-attn 1 --poll 0 -b 2048 -ub 2048`（明示 FA + poll 0）で起動し、ub=2048 単体検証と同等を確認:

| 指標 | 値 |
|------|-----|
| GPU[0] ロード | 11.64GB |
| GPU[0] 32k処理ピーク | 12.33GB（余裕 3.7GB）|
| prompt | 121.4 t/s |
| eval | 23.5 t/s |
| 131k 安定性 | OOM なし・ALIVE |

- **4 GPU の VRAM 分散**（本番構成ロード時）: GPU[0] 11.64 / GPU[1] 8.45 / GPU[2] 8.32 / GPU[3] 10.35 GB（計 ~38.8GB）。`--split-mode layer` で GPU[0]・GPU[3] に偏り、GPU[1]・[2] が低い。**GPU[0] が OOM 律速**となる。
- **prompt cache・context checkpoint も有効**: ロード時に `prompt cache is enabled, size limit: 8192 MiB`、長プロンプト処理中に context checkpoint を逐次作成（`created context checkpoint N of 32`）。同一プレフィックスの再リクエストは P100 同様に高速化される見込み。なお非 MTP モデルのため speculative decoding は当 context では無効（ログ `speculative decoding not supported by this context`）。

## P100 との比較

| 項目 | t120h-p100 | mi25 |
|------|-----------|------|
| GPU | P100 ×4 (CUDA sm_60) | MI25 ×4 (ROCm gfx900) |
| ctx | 131072 | 131072 |
| Flash-Attention | `--flash-attn 1` | `--flash-attn 1`（auto も有効）|
| batch/ubatch | `-b 4096 -ub 4096` | **`-b 2048 -ub 2048`** |
| prompt (短〜中 ctx) | 240–260 t/s | ~122 t/s |
| eval | ~12 t/s（本番）| **~24 t/s** |
| VRAM (131k) | 43.4GB / 64GB | GPU[0] ピーク 12.3GB（偏りあり）|
| llama.cpp | master 追従 | **`0fac87b15` に pin** |

mi25 は **prompt 処理は P100 の約半分**（gfx900 の演算性能・ub=2048 制約）だが、**eval（生成）は約 2 倍速い**。per-card で GPU[0] に層が偏る点が P100（均等分散で 43.4GB を 4 枚に分配）と異なる特性。

## 確定構成（start.sh 反映済み）

`start.sh` の mi25 分岐:
```bash
SERVER_OPTS="--flash-attn 1 --poll 0 -b 2048 -ub 2048"
```
KV 量子化 (`--cache-type-k/v q8_0`)・サンプリング（Qwen3.6 プロファイル `--temp 0.6 --top-p 0.95 --top-k 20 --presence-penalty 1.0 --dry-multiplier 0`）は全サーバ共通で自動適用。

## 再現方法

1. **（前提）BIOS**: MMIO High Size = 512GB、Above 4G Decoding = Enabled。`ssh mi25 'lspci | grep -c "Instinct MI25"'` が 4。
2. **モデル配置**: P100 から LAN コピー（実体 blob パスを src に指定）。
   ```bash
   .claude/skills/gpu-server/scripts/transfer-file.sh t120h-p100 \
     ~/.cache/huggingface/hub/models--unsloth--Qwen3.6-35B-A3B-GGUF/blobs/<sha> \
     mi25 ~/.cache/huggingface/hub/models--unsloth--Qwen3.6-35B-A3B-GGUF/snapshots/<hash>/Qwen3.6-35B-A3B-UD-Q4_K_XL.gguf
   ```
3. **起動**（ビルドは pin コミットで自動・gfx900 ビルド可能）:
   ```bash
   .claude/skills/gpu-server/scripts/lock.sh mi25
   .claude/skills/llama-server/scripts/start.sh mi25 "unsloth/Qwen3.6-35B-A3B-GGUF:UD-Q4_K_XL" 131072
   ```
4. **検証**:
   ```bash
   curl -sf http://10.1.4.13:8000/health
   ssh mi25 'rocm-smi --showmeminfo vram | grep Used'   # GPU[0] ~12.3GB
   # 長文プロンプトで /completion の timings を確認（prompt/eval t/s）
   ```

## 既知の課題・今後

- **llama.cpp pin の保守**: gfx900 の FP8 型問題が upstream で修正されたら（`hip.h` のアーキガード追加）、`update_and_build-mi25.sh` の `PINNED_COMMIT` を検証のうえ更新する。
- **prompt 速度の ctx 依存低下（定量）**: 32k 地点で 122.8 t/s だが、131k 近傍ではさらに低下し、**90k トークンの処理に約 25 分（≈60 t/s、32k 比ほぼ半減）**を要した（curl max-time 1500s で 90k 到達時点を計測、フル処理は未完）。gfx900 の演算性能とハイブリッドアーキの全プロンプト再処理特性による。131k フル処理は実用上重く、対話用途では中 ctx 推奨。
- **GPU[0] 偏り**: `--split-mode layer` で GPU[0] に compute buffer/logits が集中。tensor-split での均等化は ub=2048 が演算最適のため高速化に寄与せず、今回は見送り。
- **作業中に ext4 ジャーナル破損が発生**: 探索終盤、mi25 のルート FS (nvme0n1p2) が `aborted journal`・inode #2 読取り不能で read-only 再マウントされ SSH 不能となった（当初 fail2ban と誤認）。再起動で fsck 修復・rw 復帰し、4枚認識（BIOS 永続）・モデル sha256 一致を確認。**NVMe SMART は media_errors 0 / available_spare 100% / percentage_used 3% と健全**で HW 障害の証拠なし。高負荷 I/O（`rm -rf build` + 並列コンパイル `-j nproc` + 22GB コピー）が引き金の一過性論理破損と推定。再発防止としてビルド並列度を抑える・大容量コピーを分散するなどの負荷軽減が望ましい。
