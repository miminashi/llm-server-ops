## 核心発見サマリ

**修正前**: `llama-up.sh | tee log.txt` で実行すると `start.sh` の `ssh -f` が背景化後もローカル fd を保持し、`tee` が EOF を受け取れずハングする (前タスクで 3 サイクル全て観測、TaskStop 必須)。

**修正後**: 3 箇所の `ssh -f` 末尾に `</dev/null >/dev/null 2>&1` を付与しローカル fd を遮断。実機テストで `tee` 経由のパイプライン全体が `exit 0` で自然終了することを確認 (`WRAPPER_EXIT=0`、TaskStop 不要)。

| 項目 | 修正前 | 修正後 |
|------|--------|--------|
| wrapper exit | 永続ハング (TaskStop で強制終了) | `exit 0` 自然終了 |
| /health | 200 (起動自体は成功) | 200 |
| 既存挙動 | echo メッセージ等は同じ | 維持 |

## 前提・目的

- **背景**: 前タスクのレポート [`2026-05-12_105909_llama_down_unlock_order_fix.md`](2026-05-12_105909_llama_down_unlock_order_fix.md) で「引き続き残る制約」として明記。さらに先行する [`2026-05-12_051827_llama_up_down_scripts.md`](2026-05-12_051827_llama_up_down_scripts.md) でも 3 サイクル全てで観測されており、回避策として `TaskStop` 強制終了を必要としていた。

- **問題**: `.claude/skills/llama-server/scripts/start.sh` の 281, 284, 288 行で使用する `ssh -f` は背景化後も呼び出し元の stdout/stderr fd を継承するため、`llama-up.sh | tee log.txt` のパイプライン下で `tee` が EOF を受け取れずハングする。リモート側は既に `< /dev/null > /tmp/...log 2>&1 < /dev/null &` でリダイレクト済みだが、これはリモートシェル内の制御のみで、ローカル側 SSH クライアントの fd には作用しない。

- **目的**: パイプライン経由でも `llama-up.sh` がハングせず自然終了するようにする。`ssh -f` の「リモートコマンド起動完了まで return しない」保証は維持する。

- 実行ホスト: `aws-mmns-generic`

## 修正内容

### start.sh の 3 箇所に fd リダイレクト追加

`ssh -f "$SERVER" "..."` の末尾に `</dev/null >/dev/null 2>&1` を付与。SSH クライアントプロセスの stdin/stdout/stderr を `/dev/null` に向け、親プロセス (`tee` 等) との fd 継承を遮断する。

**差分** (`.claude/skills/llama-server/scripts/start.sh`):

```diff
@@ start.sh:280-288 @@
 # llama-serverをバックグラウンド起動（ssh -fでSSHを即座に返す）
-ssh -f "$SERVER" "cd ~/llama.cpp && nohup bash -c '$LAUNCH_CMD' > /tmp/llama-server.log 2>&1 < /dev/null &"
+# ローカル側 fd を /dev/null に向け、tee 等のパイプライン下でハングしないようにする
+ssh -f "$SERVER" "cd ~/llama.cpp && nohup bash -c '$LAUNCH_CMD' > /tmp/llama-server.log 2>&1 < /dev/null &" </dev/null >/dev/null 2>&1

 # ttydでログ閲覧用UIを起動
-ssh -f "$SERVER" "nohup ttyd --port 7682 --writable bash -c 'tail -f /tmp/llama-server.log' > /dev/null 2>&1 < /dev/null &"
+ssh -f "$SERVER" "nohup ttyd --port 7682 --writable bash -c 'tail -f /tmp/llama-server.log' > /dev/null 2>&1 < /dev/null &" </dev/null >/dev/null 2>&1

 # 既存nvtopプロセスを停止してttydでnvtop監視UIを起動
 ssh "$SERVER" "pkill nvtop 2>/dev/null || true"
-ssh -f "$SERVER" "nohup ttyd --port 7681 nvtop > /dev/null 2>&1 < /dev/null &"
+ssh -f "$SERVER" "nohup ttyd --port 7681 nvtop > /dev/null 2>&1 < /dev/null &" </dev/null >/dev/null 2>&1
```

### 採用しなかった代替案

- **`ssh -n + nohup &` 方式**: `ssh -f` を廃止して `&` でローカルバックグラウンド化。fd 継承問題は解決するが、`ssh -f` の「リモートコマンド起動完了まで return しない」保証を失い、wait-ready.sh との競合リスクが生じるため不採用。

### 副作用と緩和策

- **副作用**: SSH 接続失敗 (認証エラー、ホスト到達不能、リモートシェル構文エラー) の stderr が `/dev/null` に飲まれる。
- **緩和**: 起動失敗は後段の `wait-ready.sh` の `/health` 5xx/タイムアウトで検出される。診断時は `ssh "$SERVER" "cat /tmp/llama-server.log"` でリモートログを参照可能。

## 既知の制約・今後の改修

### 解消した制約

- ✅ **`start.sh` の `ssh -f` 起動による wrapper ハング** (前タスクから繰越): ローカル側 fd を `/dev/null` に向けることで解消。`tee` 等のパイプライン経由でも自然終了する。

### 引き続き残る制約 (本タスク範囲外)

- **電源 ON 後の OS 起動失敗**: SSH 待機 5 分タイムアウトで exit 1 (前タスクから据え置き)。
- **`/health` 200 だが API 異常**: 設計上「200 なら既起動」とみなし `start.sh` をスキップ (前タスクから据え置き)。
- **SSH 接続失敗時の stderr 抑制 (本修正の副作用)**: `</dev/null >/dev/null 2>&1` により SSH 認証エラー等が見えなくなる。診断時はリモートの `/tmp/llama-server.log` を確認する運用で補完。
