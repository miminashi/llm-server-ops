## 新 REPORT.md の構成

以下の見出し順に書き下ろす。

### 冒頭

```markdown
# REPORT.md

このファイルは、`llm-server-ops` プロジェクトにおけるレポート作成のルールを定める。

## レポート作成タイミング

plan mode で計画を立ててまとまった作業を行った場合、完了時に必ずレポートを作成する。

- plan mode で作業計画を立てる際、レポート作成を作業内容に含めること
- ユーザから明示的に「レポート不要」と指示された場合を除く
```

### 保存先

- `report/` ディレクトリ直下（プロジェクトルート基準）
- ワークツリーで作業している場合も、レポートは常にメインの `report/` に作成する

### ファイル名

- 形式: `yyyy-mm-dd_hhmmss_レポート名.md`
- レポート名（ファイル名部分）は英語で記載する（識別子・キーワードによる grep 性を高める）
- タイムスタンプは `TZ=Asia/Tokyo date +%Y-%m-%d_%H%M%S` コマンドで取得する（LLM が時刻を推測してはならない。システムが UTC の場合でも JST で取得される）

### レポート本文 — タイトル (H1)

- 平易な日本語で 50 字以内
- サーバ名・Unique ID 末尾 5 桁・スロット等の識別子は最小限、日時はタイトルに入れない（ファイル名側で吸収）
- 発見の詳細・数値・ハイライトはタイトルに詰め込まず、本文の「核心発見サマリ」に書く（訂正のたびにタイトルを直す必要が生じるため）
- 必要なら em dash (`—`) で結論を一行付ける形式が使える（例: `# mi25 a48e4×SLOT6 24h×2 — SLOT6 環境起因説の棄却`）

### レポート本文 — メタ情報

タイトルの直後に、以下 2 行を bullet で置く：

```markdown
- **実施日時**: 2026年7月17日 13:22 〜 13:55 JST (KVM 経由 BIOS 再設定・OS 起動確認)
- **報告日時**: 2026年7月17日 13:55 JST
```

- 日時は JST で記載する（システムが UTC の場合は +9 時間に変換）
- 「実施日時」の末尾に括弧書きで **その時間で何をしたか** の要約を付ける
- 「作成者」フィールドは使わない

### レポート本文 — 概要 (必須)

タイトルとメタ情報の直下に、他のいかなるセクションよりも先に `## 概要` を置く。

- **通読できる平易な日本語で書く**（段落として読める文章、箇条書きの羅列にしない）
- 5〜8 段落を目安
- **細かい事実（数値、識別子、パス、Unique ID、BDF、コミット SHA、Fisher 検定 p 値、trial 数、モデル名、ハッシュ等）は概要から省いてよい**。各事実は他のセクション（環境情報 / 結果詳細 / 核心発見サマリ / 参照レポート 等）に書き、概要は「何のため」「何をやった」「何が分かった」「次にやる」を平易な日本語で読める形にまとめる
- 位置づけは **「ユーザー（人間）が手短にレポート全容を把握するためのもの」** — 技術的アブスト（結論数値・p 値・BDF 等）は「核心発見サマリ」で別途書く

なお、本「概要必須」ルールは新規レポートから適用する。既存レポート（概要セクションを持たないもの）は遡及修正しない。

### レポート本文 — セクション一覧 (カノニカル名)

以下のセクションを設ける。**表記揺れがある場合は下記のカノニカル名を新規レポートで使用する**（既存レポートの書き換えは強制しない）：

| カノニカル名 | 要否 | 用途 | 併用されがちな別表現 |
|---|---|---|---|
| `## 概要` | **必須** | ユーザー（人間）が手短にレポート全容を把握できる、通読可能な平易日本語（詳細は前節） | — |
| `## 添付ファイル` | plan mode 経由なら必須 (plan.md 添付のため)、他は任意 | 生ログ・スクリーンショット・スクリプト等へのリンク | — |
| `## 核心発見サマリ` | 任意 (実測系では推奨) | 数値・PNG・統計値等を凝縮した技術者向けアブスト | — |
| `## 前提・目的` | 任意 | 背景、目的、前提条件 | 前提条件・目的 |
| `## 環境情報` | 任意 (実測系では推奨) | サーバ構成、GPU、モデル、バージョン等 | — |
| `## 参照レポート` | 任意 | 過去のレポートへの相対リンク | 過去レポート / 参考 / 参照した過去レポート |
| `## 再現方法` | 任意 | 手順・コマンド | — |
| `## 結果詳細` | 任意 | 実験結果・観測データ | 調査詳細 / 結果 / 観測データ |
| `## 副次発見` | 任意 | 副次的観測・気づき | 副次観測 |
| `## 残課題` | 任意 | 次セッションへの引き継ぎ | 現状の状態 / 既知の課題・今後 / 次セッションのタスク |
| `## 結論・対応` | 任意 (核心発見サマリで足りるなら省略可) | 結論、対応方針 | 判定 |

**表の並び順について**: 上表は「よく登場する順」に並べているが、`## 概要` を H1 とメタ情報の直下（他セクションより先）に置くこと以外、セクション順序は必須ではない。実運用では「環境情報 → 再現方法 → 結果詳細 → ... → 参照レポート」の順や、逆に「... → 参照レポート → 結果詳細」の順など、レポート内容に応じて自然な並びを選んでよい。

### レポート本文 — 核心発見サマリ の書式

「核心発見サマリ」は技術者向けアブスト（結論数値、Fisher 検定 p 値、シグネチャ照合、BDF、Unique ID 等を凝縮）。

- **実測データを持つレポートでは、同セクション冒頭に PNG を画像埋め込み `![...](...)` する**（リンクのみは不可）
- 位置は `## 核心発見サマリ` の直後・空行を挟んで最初のコンテンツ
- PNG は matplotlib 等で生成し、時系列プロットの場合は回帰直線を重畳する
- Alt テキストは図の一行説明を日本語で
- PNG 直後に `**結論**:` で始まる 1 段落テキストを続ける

コード変更のみのレポート（実測データを持たない）では PNG 埋め込みは不要。

### レポート本文 — 添付ファイル

- スクリーンショット・ログファイル・スクリプト・データ等は `report/attachment/<レポートファイル名>/` ディレクトリに格納する

# mi25 a48e4×SLOT6 24h×2 — SLOT6 環境起因説の棄却

- **実施日時**: 2026年7月5日 23:35 〜 2026年7月7日 08:00 JST (Round 1 + Round 2、累計 24h × 2)
- **報告日時**: 2026年7月10日 10:57 JST

## 概要

mi25 GPU で継続していた fault 発生問題について、原因が特定の個体にあるのか、それとも特定のスロット環境にあるのかを切り分けるための追試を実施した。過去の長時間キャンペーンでは、ある一枚のカードを問題のスロットに装着した際に fault が集中していたため、原因の切り分けが必要になっていた。

方針としては、別の個体を同じスロットに装着し、同一負荷条件で連続 2 ラウンドの長時間試験を行うこととした。個体を入れ替えても同じスロットで fault が発生するようであれば、スロット側の環境要因が疑わしいことになる。逆に fault が発生しなければ、スロット側の環境要因は否定でき、原因は個体側または個体とスロットの相互作用に絞り込める。

試験期間全体を通じて、fault は一度も観測されなかった。これにより「スロット環境そのものが fault の原因である」という仮説は棄却された。fault は個体そのものに起因するか、または特定の個体と特定のスロットの相互作用に起因するかのいずれかに絞り込めたことになる。

なお、今回の試験期間中には負荷推移や電力・温度に関するいくつかの副次的な観測も得られた。詳細は本文の該当セクションに記載している。

続くセッションでは、問題個体を別のスロットに移して同等の追試を行い、個体起因と相互作用起因のどちらかをさらに切り分ける予定である。今回の結果は、その追試の設計と mi25 の運用配置決定の材料とする。
