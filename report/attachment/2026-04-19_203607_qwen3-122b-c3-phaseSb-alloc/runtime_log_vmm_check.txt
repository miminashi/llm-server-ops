4. VMM granularity 2 MiB 仮説の再検証
---------------
2 MiB / 0.28125 MiB/tok = 7.11 tokens/block
=> 7 token ごとに 1 block 境界を跨ぐべき
   (例: 1579, 1586, 1593, 1600, ...)
   Phase Sb-fine3 で計測された ub=1586, 1588, 1592 の CUDA0 値は
   平均 0.285 MiB/tok で線形に増加 ⇒ 7 token 周期の 2 MiB step は**観測されない**

⇒ **VMM 単独での block 境界跨ぎ仮説** も単純形では**棄却**

5. 残る仮説の方向性
---------------
(i) VMM 2 MiB block + 9 層の配置が「ずれて」累積すると、
    9 層 × 0.03 MiB/tok = 0.28 MiB/tok として線形に見える
    (= 9 層が同期せず、個々の層が独立に 2 MiB 境界を跨ぐタイミングが分散)

(ii) ggml-alloc.c:303 の chunk->max_size 更新機構:
    各 tensor が chunk 内の異なる offset に配置され、max_size が
    最末尾 tensor の end で決定される。9 SSM 出力の末尾位置は
    線形に伸びる (step しない)。
    ⇒ max_size 増加は常に線形 ⇒ step は graph node の新規追加/削除で発生すべき

(iii) 但し graph nodes = 4473 固定確認済 ⇒ node 追加削除ではない

(iv) 残る可能性:
    - sched が ub=1586 で tensor を CUDA0 から他 GPU へ割り振る境界がある
      (ggml_backend_sched_split で backend assignment が変わる)
    - build_graph 内で n_tokens に応じた tensor の reshape 違いが発生
      (例: GQA batching/unbatching の閾値)
    - fused_gdn_ch (chunked) path に n_tokens ≥ 閾値 で切り替わる
      (llama-context.cpp:513 の n_tokens_ch=16*n_seqs=16 から推測、実体は要確認)

6. 結論
---------------
- VMM 有効確認 ⇒ 量子化単位は device granularity (通常 2 MiB)
- graph node/splits は ub 不変 ⇒ step は tensor 量の微差に由来
- allocator 側の簡単な量子化では step を説明できない
- 真因は **build_graph 内の ub 依存離散処理 + VMM granularity の相互作用** の可能性が高い
- 次 Phase では **debug ビルドでの backend_sched_reserve_size per-node dump** が必要
