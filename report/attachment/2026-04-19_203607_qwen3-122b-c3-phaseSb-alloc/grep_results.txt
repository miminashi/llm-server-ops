----- 候補 D (allocator pool 量子化) -----

(1) 非VMM pool: ggml-cuda.cu:422
        look_ahead_size = 256 * ((look_ahead_size + 255)/256);
    ⇒ 量子化単位 256 bytes (細かすぎて 0.28 MiB/tok step を生まない)

(2) VMM granularity: ggml-cuda.cu:L504-509
        reserve_size = granularity * ((reserve_size + granularity - 1) / granularity);
    ⇒ cuMemGetAllocationGranularity() の返す値 (P100/CC6.0 で ~2 MiB)
    ⇒ 2 MiB granularity では 1584-1600 が同じ block 486 MiB、1664 で +18 MiB 単 step
    ⇒ 観測 (1586 以降で線形 slope 0.28 MiB/tok) と不一致

(3) tensor alignment: ggml-cuda.cu:772-776
        ggml_backend_cuda_buffer_type_get_alignment() return 128;

(4) chunk max_size 更新: ggml-alloc.c:303
        chunk->max_size = MAX(chunk->max_size, offset + size);
(5) buffer size 集計: ggml-alloc.c:916-919
        new_size += new_chunk_size;

結論: 候補 D (allocator 単独の pool 量子化) は棄却。alloc_sim.py/csv で数値確証。

----- 次 Phase の方向性 -----

本 Phase で絞り込めた範囲:
    - 候補 D/E/G は棄却
    - 候補 F (graph_reserve) は step を生む容器だが機構ではない
    - 候補 H (graph splits) と 候補 I (build_graph 内 tensor shape の離散) が最有力

実測方針:
    (i) ub=1584/1585/1586/1588 の startup_log で "graph splits = N" を比較
        → splits 数が変わるなら候補 H 確定
    (ii) 同じ条件で CUDA nvprof/ncu で dst_tmp kernel の alloc サイズを記録
        → parallel_blocks の実効値が ub で変わるなら候補 I(parallel_blocks) 確定
    (iii) ggml_backend_sched_reserve_size を追加フックして、各 node の tensor size を
         ub 条件ごとに dump → ub=1586 で新規 node が現れるなら qwen35moe.cpp 側
