## ボトルネック・副次発見の分析

### 1. CUDA0 区分境界 ub\* ∈ (1584, 1600] — 本 Phase の核心発見

ub=1552 → 1568 → 1584 の 3 点は全て平坦域 slope（+0.0125〜0.0131 MiB/token）を維持し、ub=1584 → 1600 の 16-token で +4.24 MiB = 0.265 MiB/token の急増を見せる。境界 ub\* は (1584, 1600] の 16-token 区間にあり、Phase Sb-fine の 64-token 精度を **4 倍改善**。

**物理的考察**:
- 平坦域 slope 0.0125 MiB/token、境界直後 0.265 MiB/token、その後 0.2853 MiB/token。ub=1584 → 1600 の 0.265 は境界遷移中の平均値（境界より前半は平坦、後半は線形）。実際の閾値は 1584 ≤ ub\* < 1600 のいずれかに位置する整数。
- ub=1600 → 1664 の Δ=+18.26 MiB を 64 で割ると +0.2853 MiB/token、完全に線形域の slope。→ ub=1600 は既に線形域に乗っている。
- ub=1584 → 1600 の Δ=+4.24 MiB を「平坦部分」と「線形部分」に分解:
  - 仮に ub\*=1592 とすると、平坦寄与 (1584→1592, 8 token): 8 × 0.0125 = 0.10 MiB、線形寄与 (1592→1600, 8 token): 8 × 0.2853 = 2.28 MiB、合計 2.38 MiB → 実測 4.24 と合わず
  - 仮に ub\*=1585 とすると、平坦寄与 (1584→1585, 1 token): 0.013 MiB、線形寄与 (1585→1600, 15 token): 15 × 0.2853 = 4.28 MiB → 実測 4.24 と 0.96% 差、**ub\*=1585 付近の可能性大**
  - 線形のみ (ub\*=1584): 16 × 0.2853 = 4.57 MiB → 実測 +4.24 と 7.8% 差（やや超過）
- したがって**境界 ub\* は 1584 < ub\* ≤ 1588 付近（推定）**と推測できる。次 Phase で ub=1585/1586/1588/1592 の 4 点計測で 1-4 token 精度絞り込みが可能。

llama.cpp scheduler ソース (`graph_reserve` / `sched_reserve` / `llama-graph.cpp`) 側で ub=1584〜1588 付近に潜む定数（例 `n_tokens > 1584` や `n_tokens >= 1585` 等の判定）を grep する候補区間がさらに絞り込まれた。

### 2. ub=1600 セッション間再現性 — compute buffer / eval とも決定論的

Phase Sb-fine (16:16-17:00) と本 Phase Sb-fine2 (17:54-18:05) の間には約 1 時間の時間差があり、その間に Phase Sb-fine2 の Sbf2-1/2/3（ub=1552/1568/1584）の llama-server 起動・停止が挟まった。にもかかわらず ub=1600 の compute buffer 全 5 チャネルが **0.00 MiB 差で完全一致**。同条件での起動時メモリ計算は**ビット同値で決定論的**という所見。

eval 速度も 14.568 vs 14.574（Phase Sb-fine）で 0.04% 差、ほぼ同値。これは ctx=32k × ub=1600 の「境界突破直後の最遅帯」が物理的に再現性高い特性であることを示す。

### 3. ub=1584 eval サブピーク 15.293 t/s の発見

**境界直前 ub=1584 が ctx=32k 系列 17 点で 2 位の eval 速度を記録**。Phase Sb-fine の ub=1664（ピーク 15.451、境界直後）と合わせると、境界 ub\* の前後両方にピークがあり、境界そのもの ub=1600 で最遅 14.568 t/s という **「境界挟み込み型」の eval パターン** が明らかになった。

物理的解釈（仮説）:
- ub=1584 は平坦域 slope 上で CUDA0 使用量 980.11 MiB（最小領域）に収まり、GPU0 の L2 キャッシュ or memory access pattern が効率的
- ub=1600 は境界突破直後の新 staging 領域が活性化したばかりで、GPU0 パイプラインが再編成中のため非効率
- ub=1664 は新 staging 領域が安定動作して、境界突破前よりも絶対値は大きいが効率的に動作

このパターンが再現する場合、ctx=32k における **eval 最速構成候補は ub=1584 or ub=1664** の 2 つが拮抗する。prompt_tps は 4 点ほぼ同値 (±0.6%) なので、eval 重視か compute buffer 節約重視かで選択が分かれる。

### 4. Phase Sb 4p モデルの 27 点妥当性 — max_err 0.192 MiB

Phase Sb 19 点 + Phase Sb-fine 4 点 + 本 Phase 4 点 = 27 点で:
- CUDA1/2 残差 |dC1| ≤ 0.192 MiB（Phase Sb max_err 0.21 MiB と同等、Phase Sb-fine 0.185 を微更新）
- CUDA_Host 残差 |dH| ≤ 0.008 MiB（Phase Sb max_err 0.01 MiB と同等）

→ **27 点でも Phase Sb 4p モデルは R² ≥ 0.99999 を維持**し、CUDA1/2 および CUDA_Host には区分境界が**ない**ことを 4 点目の検証で再強化。**CUDA0 のみが区分的挙動**という所見は累計 27 点で確定。

### 6. eval 性能「谷山構造」の精密化 — 16-token スケールで 6 ピーク / 谷の確認

**観察**:
