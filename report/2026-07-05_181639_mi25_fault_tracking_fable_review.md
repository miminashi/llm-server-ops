## 概要

6月中旬から7月上旬にかけて、mi25 サーバの MI25 4枚のうち 1枚が負荷中に突然死する問題(GPUVM page fault)を追いかけてきました。この一連の調査は、仮説を立てて一つずつ潰していく進め方、Fisher 検定による定量判断、fault シグネチャの詳細な照合など、全体として非常に質の高いものでした。GUID が個体識別に使えないことを見抜いて Unique ID 運用に切り替えた自己訂正も見事です。

ただし、振り返ると **1つの大きな見落とし**がありました。それは「**カードとスロットがずっと同じ組み合わせだった**」ことです。6月29日に「fault はカード個体(card-c48c4)の ASIC 欠陥が原因と確定、物理交換しかない」と結論づけましたが、その根拠となった単独 24h 試験は、カードを fault 多発時と同じ SLOT6 に挿したまま行われていました。つまりあの時点の証拠では「カードが悪い」のか「SLOT6 という場所(またはカードと場所の組み合わせ)が悪い」のか、実は区別がついていませんでした。仮説の枠組み (a) VRAM 不良 / (b) 個体ロジック / (c) multi-GPU 経路 に、最初から「(d) スロット・環境要因」が入っていなかったのが根本原因です。その後スロットを移して fault が出なくなったことで、結果的にこの穴が露呈しました。過去に観測された fault 5 件はすべて「c48c4 × SLOT6」の組み合わせで起きており、SLOT8 では一度も起きていません。

もう1つの注意点は、直近レポートの「SLOT8 で fault が有意に減った (p=0.0225)」という見出しです。この比較は、スロットだけでなく GPU 枚数(4枚同時 vs 単独)も同時に変わっているため、スロットの効果だとは言い切れません。条件を揃えた比較(単独同士: SLOT8 0/221 vs SLOT6 2/147)では p=0.159 で有意差がなく、**スロット効果は統計的にはまだ実証されていません**。

幸い、この曖昧さを解消する実験は簡単です。**健全なカードを SLOT6 に挿して同じ 24h 負荷をかける**だけで、スロット環境説を直接検証できます。これはこれまでのどのレポートの残課題にも入っていなかった、最も安価な決定実験です。このほか、ディスク上のデータの再解析だけでできる宿題(fault 瞬間のテレメトリ相関、シグネチャ台帳の再監査など)や、テレメトリデーモンが試験終了後も動き続けて commit 済みログを書き換えている問題(git status の M の原因)も見つかりました。詳細は以下の各セクションに記載します。

## 発見 A: 方法論上の問題

### A-1. カード×スロットの交絡を見落として「物理交換相当」を宣言した (最重要)

[stand_alone_24h (2026-06-29_041700)](2026-06-29_041700_mi25_8820_stand_alone_24h.md) は「単独可視化でも fault が再現した → (c) multi-GPU 経路を否定、(b) 個体ロジック起因**確定**、物理交換相当」と結論した。(c) の否定は正当である。しかし **(b) の「確定」は当時の証拠では正当化されない**:

- 単独 24h 試験は、カードを **4枚運用時の fault と同じ SLOT6 (BDF 87:00.0) に挿したまま**実施された。カード個体とスロットは一度も分離されておらず、「カード起因」と「SLOT6 環境起因(信号品質・電源系統・接点状態、あるいは個体×スロット相互作用)」は観測上等価だった
- 仮説枠組み (a) VRAM bad page / (b) 個体ロジック / (c) multi-GPU 経路 には、**(d) スロット・環境要因/相互作用が最初から存在しなかった**。枠組みにない仮説は消去法で残らないため、(a)(c) を潰した時点で (b) が「確定」してしまう構造だった
- 全履歴を通じた fault 実績は: **c48c4×SLOT6 で 5/235** (4枚 3/88 + 単独 2/147)、**c48c4×SLOT8 で 0/258** (8h 37 + 24h×2 221)。このデータは (b) 単独起因とも (d) スロット/相互作用起因とも整合する
- 「物理交換相当」「4枚 64GB には物理交換必須」の断定は CLAUDE.md とメモリにも伝播しており、07-04 の結果を受けた現在は**過大な断定**になっている (推奨 D-5)

なお、翌日の [slot_move_load (06-30)](2026-06-30_012759_mi25_c48c4_slot_move_load.md) が「fault が Unique ID に追従するか」を検証しに行ったこと自体は、この交絡に事後的に気づいた正しい動きである。問題は 06-29 時点で「確定」「物理交換相当」という不可逆な意思決定語彙を使ったことにある (実際に物理交換を発注していれば無駄になった可能性がある)。

### A-2. 07-04 の見出し p=0.0225 は多重に交絡した比較

[slot8_24h_x2 (07-04)](2026-07-04_012209_mi25_c48c4_slot8_24h_x2.md) の統計的ハイライト「vs 過去 4枚運用 3/88 で p=0.0225、5% 水準で有意に fault 低下」は、**スロット (SLOT6→SLOT8) と GPU 枚数 (4枚同時→単独可視化) を同時に跨ぐ比較**である:

- stand_alone_24h 自身が「発火率低下 1.36% vs 3.41% は (c) multi-GPU 経路の微小寄与の可能性を残す (p=0.27)」と明記している。つまり 3.41%→0% の低下は「枚数効果 + スロット効果」の合成であり、p=0.0225 をスロット効果の証拠として読むことはできない
- **条件を揃えた比較** (単独可視化同士: SLOT8 0/221 vs SLOT6 2/147) は p=0.1589 で非有意 — この数値はレポート本文にも記載されているが、見出し・コミットメッセージには反映されていない
- 0/221 の 95% 信頼上限は rule of three で ≈ 3/221 ≈ **1.36%** であり、SLOT6 単独の実測率 1.36% と一致する。すなわち **データは「SLOT8 でも SLOT6 単独と同じ fault 率」という帰無仮説と完全に整合**しており、スロット効果は統計的に未実証
- レポートは少なくとも 3 つの Fisher 比較を並置しているが多重比較補正がない (Bonferroni なら 0.0225×3 = 0.0675 > 0.05 で有意性消失)
- コミットメッセージ「221 trial / 0 fault で **(b) を SLOT8 で否定**」は、本文のヘッジ表現 (「否定される方向」) より強く、上記の限定を落としている。git log だけを読む将来のセッションには過大な結論が伝わる

公平のため付記: レポート自身は P(0)≒4.85% の検出力計算を示し、「SA 水準 1.36% は棄却できない」と正しく明記している。問題は本文の慎重さと、見出し・コミットの断定の乖離である。

### A-3. 未解明の異常が結論と同居している

- **pp_tps の半減**: SLOT8 系試験の prompt 処理は 206-230 t/s で、SLOT6 単独時の 508.9 t/s の 40-45% しかない (eval_tps は 23.8-24.4 vs 23.5 で完全整合)。同一カード・同一モデル・同一パラメータで pp だけが半減するのは「同等構成での比較」という前提を損なう構成差シグナルだが、原因未追究のまま fault 比較の結論が出されている。fault 発火が prompt 処理経路の負荷に依存する場合、SLOT8 試験は実質的に曝露が弱かった可能性を排除できない
- **R1/R2 間の電力差**: 同一条件のはずの Round 1/2 で mean power が 114.6W vs 95.6W と 17% 異なり、説明がない

### A-4. 電力・熱曝露が試験間で統制されていない

- SLOT6 単独試験は R2 trial 24 の BACO reset で power cap が 220W (HW default) に戻ったまま trial 25-113 の 89 trial を走らせた (レポート自身が「再設定する余裕がなく fault 後はそのまま継続」と記録)。結果として SLOT6 単独 = p95 185W / max 225W / Tj max 99°C、SLOT8 = p95 161W / Tj max 95°C と、比較対象間で電力・熱曝露が異なる
- ただし SA 内部の証拠は電力主因説に不利であることも公平に記す: **fault 2 件はいずれも 160W cap 期に発生し、220W 期の 89 trial は 0 fault** だった。また 4枚時代の per-card 実消費は 36-39W に過ぎず (sweep レポート)、低電力でも 3/88 発火している。電力・熱は主因である可能性が低いが、「統制されていない」事実は SLOT6/SLOT8 比較の残余交絡として残る
- stand_alone_24h が自ら宿題とした「**fault 発生時の Tj 値との相関解析**」は、その後どのレポートでも実施されていない。必要なデータ (fault 時刻 + telemetry_rocmsmi) はすべてディスク上にある

### A-5. fault シグネチャ台帳に未監査の曖昧さがある

シグネチャ照合 (SQC→CB バースト順序の一致) 自体は本追跡で最も質の高い分析の一つだが、**「過去 4枚 3/88 と完全同一シグネチャ」という台帳の帳尻には監査されていない曖昧さ**がある:

- stand_alone_24h の dmesg リングバッファ解析では、過去 fault バーストは uptime 29459 / 41888 / 43173 の 3 件。うち no-retry page fault バーストとして確認されたのは **2 件** (29459 / 41888、レポート内でも「uptime 29399 / 41827 の 2 件」と照合) で、**3 件目 (43173) は「06-25 4枚復旧の負荷検証 (ROCm 経路、no-retry page fault を伴わない)」と注記されている**
- 一方、uptime 間隔の算術はこの由来ラベルと緊張関係にある: 29459→41888 の差 12429s はスイープ 4 点分 (点あたり ~3000s) とよく整合し 175W→155W fault に対応するが、41888→43173 の差 1285s は「06-25 の別試験」よりむしろ **150W fault (t2f=1515s、fault で phase 早期終了と仮定)** に近い。もし 43173 が 150W fault なら「3 件目のシグネチャは no-retry を伴わない別型」となり、**「3 件同一」主張か由来ラベルのどちらかに誤りがある**
- いずれであっても結論の大勢 (c48c4×SLOT6 に fault が集中) は揺らがないが、シグネチャ完全一致は (b) 確定の主要根拠だったため、一次データでの再監査を推奨する (推奨 D-3)
- 関連して、**fault アドレスの解析が一度も行われていない**。ROCm fault は `0x100000000` (ちょうど 4GiB 境界 = bit 32 のみ立ったアドレス)、Vulkan fault は `0x2c000` (低位アドレス)。4GiB 境界はアドレス経路のビット異常やアパーチャ境界問題を示唆しうる特徴的な値であり、VM_L2_PROTECTION_FAULT_STATUS のビットデコード (0x00801031 = TCP client vs 0x004012B1 = WALKER_ERROR/MAPPING_ERROR 付き) と合わせて、5 件の fault を体系的に比較する再解析の余地がある

### A-7. 最も安価な決定実験が提案リストに存在しない

現在の残課題リスト (slot8_24h_x2) は「c48c4×SLOT8×4枚同時 24h」を挙げているが、A-1 の交絡を直接解消する実験 — **健全カード (a48e4 / c3164 / 448c4 のいずれか) を SLOT6 に挿して単独 24h 負荷** — はどのレポートでも提案されていない。

## 発見 C: 良かった点

公平のため、本追跡の水準の高さも記録する:

- **仮説駆動の系統的消去法**: (a)(b)(c) を memtest 1.29PB / 単独 24h / 追試スイープで一つずつ潰す設計は教科書的。電力点固有説を「完全同一条件の追試 44/44」で否定した pwr_sweep v2 は特に厳密
- **統計の常用**: Fisher 検定・P(0) 検出力・信頼区間が一貫して使われ、slot_move_load (8h/37 trial) では「P(0)≒60% なので判定保留」と正しくサンプル不足を宣言している
- **fault シグネチャ法医学**: vmid の動的差を除いた ring/pasid/BDF/UTCL2 client バースト順序 (SQC→CB×8-9) の照合は説得力が高い
- **自己訂正の文化**: GUID 非不変の発見 (06-29)、SLOT 番号の再訂正 (06-30、自らの前日の訂正を SMBIOS 実照合で覆した)、ハーネスバグ 3 件 (偽終了・restart 忘れ・dmesg 履歴混入) の率直な文書化と修正
- **再利用可能な運用資産**: dmesg baseline 行数フィルタ、/health pre-check + auto-restart、BMC スクショによる証跡保全ルール、rc.local の power cap 永続化解明

## 発見 D: 推奨アクション (優先順)

1. **健全カード×SLOT6 単独 24h 負荷** (A-7)。2×2 マトリクスの空セルを埋める最安の決定実験。147 trial 水準なら 1.36% に対し P(≥1)≒87%
2. **c48c4×SLOT8×4枚同時 24h** (既存残課題#2)。SLOT8 運用採否の前提条件
3. **ディスク上データの再解析のみで可能な宿題** (サーバ時間ゼロ): fault 瞬間の telemetry 相関 (A-4)、シグネチャ台帳の再監査 — 特に uptime 43173 の由来 (A-5)、fault アドレス・FAULT_STATUS ビットの 5 件比較 (A-5)
4. **VBIOS / RAS カウンタの 4枚比較** (A-6)。サーバ数分で完了
5. **CLAUDE.md・メモリの断定表現の更新**: 「(b) 個体ロジック起因確定・物理交換相当/必須」→「fault は c48c4×SLOT6 の組み合わせでのみ観測 (5/235)、SLOT8 では未観測 (0/258)。個体起因かスロット/相互作用起因かの弁別は未完 (推奨 1・2 の結果待ち)」
6. **運用是正**: 残存テレメトリデーモン 2 系統の停止 → 添付ログ 6 ファイルの扱い決定 (追記分を破棄して commit 時点へ restore するか、追記込みで再 commit するか) → `MNL-1677.pdf` / `mb_slots_zoom.png` の attachment への整理 → push。ハーネスには H-1 (正常完了時の自動停止) の実装を推奨

## 結論

本追跡は消去法の設計・統計の運用・証跡保全のいずれも高水準で、(a) VRAM bad page と (c) multi-GPU 経路単独起因の否定、および c48c4×SLOT6 での fault 再現性の確立は堅牢である。一方で、**仮説空間に「スロット/環境要因」が欠けていたため、カードとスロットが交絡したまま「個体起因確定・物理交換相当」と宣言した**のが最大の方法論的見落としであり、07-04 時点の「SLOT8 で有意に抑制 (p=0.0225)」も枚数効果と混ざった比較であるため、**「カードが悪い」のか「c48c4 と SLOT6 の組み合わせが悪い」のかは現時点でも未決着**である。健全カード×SLOT6 の 24h 試験 (推奨 1) が、この 3 週間の追跡を完結させる欠けた 1 ピースである。
