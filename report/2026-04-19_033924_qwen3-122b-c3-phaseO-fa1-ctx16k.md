### 2. 核心発見：fa=1 の compute buffer は ctx=8192 以降で頭打ち

Phase N までで得た 4 点フィット（ctx=1024〜8192）による ctx=16384 予測値と、Phase O 実測値の差分:

| GPU | 4 点フィット予測 @ctx=16384 | Phase O 実測 | 予測 − 実測 |
|----|---:|---:|---:|
| CUDA0 | 5,295.3 | **2,784.0** | **+2,511.3 MiB** |
| CUDA1 | 4,160.5 | **2,080.3** | **+2,080.2 MiB** |
| CUDA2 | 4,160.5 | **2,080.3** | **+2,080.2 MiB** |
| **CUDA3** | **16,096.0** | **8,048.0** | **+8,048.0 MiB** |
| CUDA_Host | 1,408.6 | 704.3 | +704.3 MiB |
| 合計 | 31,120.9 | **15,696.8** | **+15,424.1 MiB** |

**全 GPU で実測は予測の約半分**。特に CUDA3 は ctx=8192 と ctx=16384 で **完全同一の 8,048 MiB**。

| ctx | CUDA3 実測 | 線形仮説 (b=0.9824×ctx) |
|---:|---:|---:|
| 1,024 | 1,006.00 | 1,006.00 (完全一致) |
| 2,048 | 2,012.00 | 2,012.00 (完全一致) |
| 4,096 | 4,024.00 | 4,024.00 (完全一致) |
| 8,192 | 8,048.00 | 8,048.00 (完全一致) |
| **16,384** | **8,048.00** | **16,095.64（理論値より 50% 少ない）** |

**CUDA3 は `min(ctx, 8192) × 0.9824` に従う**ことが強く示唆される。**決定因子は ctx ではなく `-b 8192`（バッチサイズ）**。ctx ≤ -b では compute buffer ≈ ctx × 0.9824、ctx > -b では compute buffer ≈ -b × 0.9824 で飽和。

### 3. 5 点フィットは「2 次多項式モデル自体が不適切」と判定

ctx=16384 データを加えた 5 点フィット結果:

| GPU | a | b | c | max resid | log-log k |
|----|---:|---:|---:|---:|---:|
| CUDA0 | -7.97e-6 | 0.266 | 581.88 | 129.20 | 0.425 |
| CUDA1 | -1.38e-5 | 0.366 | -203.95 | 119.73 | 0.843 |
| CUDA2 | -1.38e-5 | 0.366 | -203.95 | 119.73 | 0.843 |
| CUDA3 | -6.07e-5 | 1.537 | -789.02 | **463.23** | 0.800 |
| CUDA_Host | -1.50e-6 | 0.072 | -69.05 | 40.54 | 1.211 |
| 合計 | -9.77e-5 | 2.607 | -684.08 | 852.57 | 0.722 |

- **全 GPU で a 係数が負に転じる**（Phase N 4 点では CUDA1/2 a=1.91e-6 が正）
- max resid は CUDA3 で 463 MiB と大幅に悪化（Phase N 4 点 0 MiB → Phase O 5 点 463 MiB）
- **CUDA0 の resid@16384 は -15.88 MiB で小さい**が、合計の max resid が 852 MiB と巨大
- log-log k は CUDA3=0.800、CUDA1/2=0.843 に低下（Phase N の 1.00/1.028 から大幅乖離）

**結論**: 2 次多項式モデル `a·n² + b·n + c` は ctx=8192 以降で飽和する実測を捉えられない。**区分モデル `min(ctx, -b) に対する 2 次多項式` の導入が必要**。

### 4. 正しいモデル構造：バッチサイズ支配

compute buffer の真のドライバは **`n_eff = min(ctx, -b=8192)`** と推定される:

```
fa=1 compute_buffer(ctx, b=8192, ub=8192):
  n_eff = min(ctx, b)
  CUDA0:    (未特定、c=828 + linear*n_eff 程度、定数項支配)
  CUDA1/2:  1.91e-6 * n_eff^2 + 0.2227 * n_eff    (Phase N の係数で n_eff を使用)
  CUDA3:    0.9824 * n_eff                         (ctx=16384 で完全一致確認)
  CUDA_Host: 3.81e-6 * n_eff^2 + 0.0235 * n_eff    (比例的)
```

検証（ctx=16384 を `n_eff=8192` として評価）:

| GPU | Phase N 4pt モデル @ n_eff=8192 | Phase O 実測 @ ctx=16384 | 誤差 |
|----|---:|---:|---:|
| CUDA0 | 2,320.5 | 2,784.0 | +463.5 (+20%、c 項変動の可能性) |
| CUDA1/2 | 1,952.3 | 2,080.3 | +128.0 (+6.6%) |
| CUDA3 | 8,048.0 | 8,048.00 | **0.0 (完全一致)** |
| CUDA_Host | 448.3 | 704.3 | +256.0 (+57%、再精査要) |

CUDA3 のみ完全一致、CUDA1/2 は 6% 増、CUDA0 は 20% 増、CUDA_Host は 57% 増。**CUDA1/2/3 は `n_eff=min(ctx, -b)` モデルで概ね説明可能**だが、**CUDA0 と CUDA_Host は ctx 全域に依存する成分が残存**（具体的にはポジションエンコーディングや RoPE の作業領域）。

## ボトルネック・副次発見の分析

### 1. Phase N 4 点モデルは「2 次多項式の外挿限界」が ctx=8192 であることの証明

Phase N では CUDA1/2 の log-log k=1.028 を「ほぼ完全線形」と解釈していたが、Phase O の 5 点目 (ctx=16384) を加えると **予測値は実測の 2 倍**。これは「ctx=8192 までは線形増加、ctx > 8192 では頭打ち」という **区分モデル** の最初の証拠。

4 点モデルの外挿性能は「**データ範囲内（ctx ≤ 8192）では極めて正確、範囲外では物理的にあり得ない外挿値**」という一般則が fa=1 側でも成立することが確認された（Phase M で fa=0 側で観察した「ctx=16384 で 145,584 MiB」と類似）。

### 2. `-b 8192`（バッチサイズ）が compute buffer の真のドライバ

起動ログ `graph splits = 136 (with bs=8192), 77 (with bs=1)` から、**graph の構造は batch size 単位で決定**されている。CUDA3 の完全一致（ctx=8192, 16384 で 8048 MiB）から、**compute buffer は `graph` のサイズ（bs=8192 で 136 split）に依存し、ctx には依存しない**ことが強く示唆される。

これは llama.cpp の実装上の「**compute buffer = 1 回の forward pass で必要なテンソルの総和**」で、1 回の forward pass で処理するトークン数は `ub` または `b` で制限されるため、ctx > -b では compute buffer が飽和する。

### 3. CUDA0 / CUDA_Host は ctx 全域に依存する成分が残存

| GPU | n_eff=8192 モデル予測 | ctx=16384 実測 | 誤差 |
|---|---:|---:|---:|
| CUDA0 | 2,320 | 2,784 | +463 MiB (+20%) |
| CUDA_Host | 448 | 704 | +256 MiB (+57%) |

CUDA1/2/3 は `n_eff=min(ctx, -b)` モデルでほぼ完璧に予測できるが、CUDA0 と CUDA_Host には **ctx 全域に依存する成分**（おそらく位置埋め込み、RoPE テーブル、入力 token IDs の buffer 等）が残存。

CUDA0 は embedding 層 + output head を持つので、出力 logits 行列（ctx に比例）や入力 token embedding の buffer が compute 側でも必要となる可能性。

### 4. Phase K の eval 15.046 t/s との 0.23% 差は計測内ゆらぎ

Phase K は warmup 1 run のみの採取だったが、Phase O の 3 run で平均 15.007、中央値 15.011 t/s。**Run 間 range 0.030 t/s** に収まっており、Phase K 値 15.046 はこの range の上辺ちょうど。**セッション間の計測ゆらぎは 0.3% 以下で制御可能**であることが確認された。

### 5. graph nodes=4473 は全 fa=1 ctx 条件で同一

ctx=1024/2048/4096/8192/16384 のすべてで `graph nodes = 4473` かつ `graph splits = 136 (with bs=8192)` が同一。これは **graph 構造は ctx に依存せず、batch size のみに依存**することを示す。ctx の違いは graph の各ノードで処理される **テンソル次元のサイズ**（attention の KQV 行列の n×n 部分）に反映されるが、ノード数自体は不変。

## 採用判定

| 項目 | 結果 |
|------|------|
| fa=1 ctx=16384 起動 | ✅ 成功（PID=164760、Phase K 再現） |
| sched_reserve 採取 | ✅ 4 GPU + CUDA_Host の全値取得 |
| 5 点フィット結果 | ❌ **2 次多項式は ctx=16384 で適合せず**（CUDA3 で resid 463 MiB） |
| モデル構造の訂正 | ✅ `n_eff = min(ctx, -b)` 区分モデルが必要と判明 |
| CUDA0 非線形性の改善 | ❌ **改善せず**（4 点 resid 70 → 5 点 129）、ただし原因は ctx=16384 の頭打ちと判明 |
| Phase K 再現性 | ✅ eval 15.011 vs 15.046 t/s（-0.23%、Run 間 range 内） |
| C-D3 採用構成の妥当性 | **強化**（ctx ≥ 8192 では eval / compute buffer / GPU mem がすべて ctx 非依存） |

**結論**: Phase O は「5 点フィットで CUDA0 の 70 MiB resid を解消する」という当初目的を果たさなかったが、**より本質的な知見**（fa=1 の compute buffer は `-b=8192` 以降で飽和、真のドライバは `n_eff=min(ctx, -b)`）を獲得した。本番 `start.sh` の改変は不要だが、compute buffer 予測モデルは **区分モデル化** が必要。

## 補足

### Phase O の核心発見

1. **fa=1 の compute buffer は `-b=8192` で頭打ち** — ctx=16384 でも compute buffer 合計 15,697 MiB（ctx=8192 の 14,721 MiB の +7% のみ）
2. **CUDA3 は `min(ctx, -b) × 0.9824` で完全予測** — ctx=8192 と ctx=16384 で CUDA3=8,048 MiB の完全一致
3. **Phase N の 4 点フィット外挿は ctx=16384 で 2 倍誤差** — 全 GPU で予測が実測の 2 倍、2 次多項式モデルの外挿限界は ctx=8192 側境界
4. **graph nodes=4473、graph splits=136 は ctx に依存しない** — graph 構造は batch size 単位で決定
5. **CUDA0 / CUDA_Host は ctx 全域依存成分が残存** — 位置埋め込み or 出力 logits 関連の可能性
6. **Phase K との eval 再現性 0.23%** — セッション間ゆらぎは十分制御可能

### 計算モデルの決定版（f16 KV, C-D3 base, -b=8192）

```
fa=0: compute_buffer(n)  [単位 MiB, n = ctx, ctx ≤ 4096 でのみ検証済]
  CUDA0:    2.454e-4 * n^2 - 0.682 * n + 1562.67
  CUDA1/2:  1.259e-4 * n^2 + 0.13  * n
  CUDA3:    0.982    * n
  CUDA_Host: 3.82e-6 * n^2 + 0.027 * n

fa=1: compute_buffer(n)、n_eff = min(n, -b=8192) を使用
  CUDA0:    828 + 0.09 * n_eff + [ctx 依存残差 0〜463 MiB]
  CUDA1/2:  1.91e-6 * n_eff^2 + 0.2227 * n_eff   (ctx=16384 で +6.6% 誤差)
  CUDA3:    0.9824   * n_eff                      (完全一致、Phase O 実証)
  CUDA_Host: 3.81e-6 * n_eff^2 + 0.024 * n_eff + [ctx 依存残差 0〜256 MiB]
```

### fa=1 の起動可能範囲（P100 16GB × 4 構成、拡張）

| ctx | compute buffer 合計 | 主要 bottleneck |
|---:|---:|---|
| 1,024 | 2,469 MiB | CUDA3 |
| 2,048 | 3,998 MiB | CUDA3 |
| 4,096 | 7,500 MiB | CUDA3 |
| 8,192 | 14,721 MiB | CUDA3 |
| 16,384 | **15,697 MiB** | CUDA3 (頭打ち) |
| 32,768 (予測) | **≈ 15,700 MiB** | CUDA3 (頭打ち継続予測) |
| 131,072 (本番) | **≈ 15,700 MiB** | CUDA3 (頭打ち継続予測) |

ctx > 8192 では compute buffer が増えないため、**本番の ctx=131k でも compute buffer は ctx=16384 と同等の 15,700 MiB と予測**。これは「なぜ 131k が起動可能か」の直接的説明。
