## 前提・目的

- **目的**: 前身レポートで「本質的な最適化は両ノードを使いつつ NUMA リモートアクセスを減らす方向」と結論されたため、その候補 2 方式（`numactl --interleave=all` / `--numa distribute`）を C-C1（現行 Phase B 構成）と比較し、eval 速度改善を検証する
- **仮説**:
  - **C-C2** (`numactl --interleave=all`): メモリをラウンドロビンで両ノードに配置し、80 論理 CPU を活用。N0/N1 ともにローカルヒット率が上がり、前身 Phase B（N1 限定 40 CPU）よりスループット向上
  - **C-C3** (`--numa distribute`): llama.cpp 側のスレッド分散で NUMA-aware 配置。ただしメモリは OS 任せ
- **成功条件**: C-C2 / C-C3 いずれかが C-C1 中央値 × 1.03 以上
- **観測負荷最小化**: 前身レポートで perf / mpstat / pidstat の並列実行が観測負荷の源と推定されたため、本計測では **perf stat / record・mpstat・pidstat・numastat を使用しない**（C-C1 の Run 2 のみ参考として perf stat を 10 秒間取得）

## ボトルネック分析

### C-C1 が前身 Phase B (11.50 t/s) から 7.41 t/s に低下した原因

前身レポートの Phase B では `numactl -N1 -m1 + --threads -1` で 11.50 t/s を計測したが、同一構成の fresh restart で 7.41 t/s に低下。

**原因**: `--threads -1` は `sysconf(_SC_NPROCESSORS_ONLN)` から **n_threads=80** を決定する。`numactl --cpunodebind=1` は Cpus_allowed_list を 40 CPU に制限するが、llama.cpp はこれを参照せず 80 スレッドを起動する。80 スレッド ÷ 40 CPU = **2 倍のオーバーサブスクリプション**。

前身レポート計測時は perf record / mpstat / pidstat / numastat が並列稼働しており、これらの観測ツール群が CPU 時間を消費した結果、実効的なワーカースレッドの同時稼働数が減少し、オーバーサブスクリプションの悪影響が偶然緩和された可能性がある。観測負荷を除去した本計測で真のスレッド競合コストが露呈した形。

voluntary_ctxt_switches の差（前身 72,108 vs 本計測 227,304）も、前身計測が短時間（40 秒 eval × 3 run）で本計測が 3 run × 60 秒 cooldown を含むより長い稼働期間であることが一因。

### C-C2 が最速である理由

1. **80 CPU 全活用**: Cpus_allowed_list=0-79 で 80 スレッド × 80 CPU = 1:1 マッピング
2. **メモリ interleave**: `--interleave=all` でページ配置がラウンドロビン。N0 CPU は N0 ローカルメモリに、N1 CPU は N1 ローカルメモリにアクセスでき、NUMA リモートアクセスが統計的に半減
3. **低コンテキストスイッチ**: voluntary=854 でスレッドが効率的にビジーループ
4. **ページキャッシュの偏在**: モデルファイルが前セッションから Node 1 のページキャッシュに載っていたため、interleave は完全均等にならず N0=3.9%。**完全にコールドな状態で起動すれば N0/N1 が約 50:50 に分散し、さらなる改善の余地がある**

### C-C3 が C-C2 に劣る理由

- スレッドは両ノードに分散するがメモリは OS 任せ（Node 1 に 99.9% 偏在）
- Node 0 のスレッドは Node 1 のメモリに NUMA リモートアクセス
- C-C2 の interleave でこの問題が軽減されるため C-C2 が優位

```bash
# 1. ロック取得
.claude/skills/gpu-server/scripts/lock.sh t120h-p100

# 2. 既存プロセス停止
.claude/skills/llama-server/scripts/stop.sh t120h-p100

MODEL_PATH="/home/llm/.cache/huggingface/hub/models--unsloth--Qwen3.5-122B-A10B-GGUF/snapshots/51eab4d59d53f573fb9206cb3ce613f1d0aa392b/Q4_K_M/Qwen3.5-122B-A10B-Q4_K_M-00001-of-00003.gguf"

LAUNCH_CMD="numactl --interleave=all -- ./build/bin/llama-server \
  -m '$MODEL_PATH' --jinja \
  -ngl 999 -ot 'blk\\.([0-9]|1[0-3]|2[0-4]|3[1-9]|4[0-7])\\.ffn_.*_exps\\.weight=CPU' \
  --flash-attn 1 --poll 0 -b 8192 -ub 8192 \
  --n-predict 32768 --threads -1 \
  --ctx-size 131072 --parallel 1 \
  --cache-type-k q8_0 --cache-type-v q8_0 \
  --temp 0.6 --top-p 0.95 --top-k 20 --min-p 0 \
  --port 8000 --host 0.0.0.0 \
  --alias 'unsloth/Qwen3.5-122B-A10B-GGUF:Q4_K_M'"

ssh -f t120h-p100 "cd ~/llama.cpp && nohup bash -c \"\$LAUNCH_CMD\" > /tmp/llama-server.log 2>&1 < /dev/null &"

# 4. ヘルスチェック
until curl -sf http://10.1.4.14:8000/health > /dev/null; do sleep 5; done

# 5. ロック解放
.claude/skills/gpu-server/scripts/unlock.sh t120h-p100
```
