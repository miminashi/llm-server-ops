# mi25 Vulkan で Qwen3.6-35B-A3B 128k（pin不要・ub非依存）

- **実施日時**: 2026年06月14日 00:11 (JST)

## 添付ファイル

- [実装プラン](attachment/2026-06-14_001107_mi25_vulkan_qwen36_128k/plan.md)

## 核心発見サマリ

![ub 掃引: GPU[0] VRAM と速度（Vulkan vs ROCm）](attachment/2026-06-14_001107_mi25_vulkan_qwen36_128k/ub_fa_sweep.png)

mi25 (MI25 ×4, gfx900) で Qwen3.6-35B-A3B (UD-Q4_K_XL) を **Vulkan (RADV) バックエンド・128k (131072) で安定動作**させ、ROCm 版（[既存レポート](2026-06-13_112006_mi25_qwen36_128k.md)）と同じパラメータ探索を行った。ROCm とは性質が大きく異なる4点を確認した。

1. **Vulkan は llama.cpp 最新 master を pin なしでビルドできる** — ROCm は `__hip_fp8_e4m3` 型問題でコミット `0fac87b15` (v8533) に pin が必須だが、Vulkan は HIP の device コンパイルを伴わないため当該リグレッションに無関係。**master `57fe1f07c` (v9620) をそのままビルド成功**（ROCm pin より 1000+ コミット新しい）。`build-vulkan/` に分離ビルドし ROCm の `build/` と共存。

2. **prompt 処理が ROCm の約3.3倍** — 32k で **Vulkan ~400 t/s（ub 2048〜4096 で 401〜405）vs ROCm 122.8 t/s**。100k 近傍でも **Vulkan 216 t/s vs ROCm ~60 t/s（約3.6倍）**。一方 **eval（生成）は ROCm 24.5 t/s に対し Vulkan ~15 t/s（約0.6倍）** と逆転。Vulkan は prefill 特化、ROCm は decode 特化という明確な棲み分け。

3. **ub を上げても VRAM・速度がほぼ不変（ROCm の OOM 問題が消失）** — ROCm では ub=4096 で GPU[0] が 16.8GB に達し 32k で OOM クラッシュしたが、**Vulkan は ub 2048/3072/4096 のいずれでも GPU[0] が 8.72GB で完全一定**・prompt も ~405 t/s 一定。RADV では FA の compute buffer が ubatch に比例しないため、ub 上げによる VRAM 肥大も速度低下も起きない。

4. **VRAM 使用が低く均等** — Vulkan は GPU[0] 8.72 / [1] 8.22 / [2] 8.09 / [3] 8.34 GB（計~33GB）と均等分散。ROCm の GPU[0] 偏り（11.6〜16.8GB）と対照的で、16GiB（17.16GB）に対し各 GPU 8GB 以上の余裕がある。

**確定構成（Vulkan）**: `--flash-attn 1 --poll 0 -b 2048 -ub 2048`（KV q8_0、ctx 131072、`GGML_VK_VISIBLE_DEVICES=0,1,2,3`）。
ub は性能に影響しないため ROCm と同じ ub=2048 を踏襲（eval が僅差で最良）。128k 範囲（100k トークン処理）で **OOM なし・VRAM 一定・ホスト安定**を実証。

> **注意（高負荷時の不安定性）**: 探索中、KV を **f16** に変えた FA=1 トライアルの **32k 推論実行中**に mi25 が SSH/ping 不能となりダウンした（推論リクエストが空応答で返り、直後の次トライアル起動で `No route to host`。ユーザによる電源再投入で復旧。再起動後 root FS は rw 健全・GPU 4枚認識・BIOS MMIO 設定維持）。なお **f16 KV は VRAM 的には収まっていた**（ロード時 GPU[0] 8.97GB、q8_0 比 +約0.25GB）ため、ダウンはメモリ不足ではなく安定性問題である。q8_0 構成では 100k 処理を含め多数の試行で完全に安定していたため、f16 KV + FA の RADV シェーダ経路、または高負荷 I/O（[既存レポートの ext4 一過性破損事例](2026-06-13_112006_mi25_qwen36_128k.md)と同種）が引き金と推定。根本原因は未確定だが、**本番は実証済みの q8_0 構成を使用し f16 KV は避ける**。

## 前提・目的

- **背景**: ROCm 版 mi25 は llama.cpp を gfx900 ビルド可能コミットに pin する保守コストを抱える（[既存レポート](2026-06-13_112006_mi25_qwen36_128k.md)）。Vulkan ビルドは HIP を伴わないため pin が不要なはずで、保守性・性能の両面で代替候補になりうる。
- **目的**: mi25 を Vulkan (RADV) でビルドし、ROCm 版と同じ探索（ub 掃引 + FA 有無、128k/KV q8_0）を実施。pin 不要を実証し、ビルドスクリプトを Vulkan 対応に拡張する。
- **前提条件**: mi25 が利用可能。Vulkan ツールチェーン（RADV ICD・glslc・vulkan-headers）が導入済みであること。

## 環境情報

- **サーバ**: mi25 (10.1.4.13)
- **GPU**: AMD Radeon Instinct MI25 ×4（gfx900 / Vega 10、VRAM 16GiB/枚、計 64GB）
- **Vulkan ドライバ**: Mesa RADV（`radeon_icd`）、Vulkan Instance 1.4.313。4枚すべて `Radeon Instinct MI25 (RADV VEGA10)` として列挙（加えて `llvmpipe` (CPU) も列挙されるため除外が必要）
- **ビルド依存**: `glslc` (shaderc v2023.8)、`vulkan-headers` 1.4.313、**`spirv-headers` 1.6.1 / `spirv-tools` 2024.4（本作業で追加インストール）**。llama.cpp の Vulkan ビルドは `find_package(SPIRV-Headers CONFIG REQUIRED)` を要求する
- **llama.cpp**: `57fe1f07c` (**version 9620**) — Vulkan は master 追従（pin なし）。ROCm 版の pin `0fac87b15` (v8533) と共存
- **モデル**: `unsloth/Qwen3.6-35B-A3B-GGUF:UD-Q4_K_XL`（22.36GB）
- **KV 量子化**: `--cache-type-k/v q8_0`（全試行）
- **比較対象**: ROCm/HIP 版 mi25（[既存レポート](2026-06-13_112006_mi25_qwen36_128k.md)）

## Phase 0: Vulkan ビルド（pin 不要の実証）

`update_and_build-mi25.sh` に `MI25_BACKEND` 分岐を追加し、`MI25_BACKEND=vulkan` で実行:

- `git checkout master && git pull` で **最新 master を取得**（pin なし）→ `57fe1f07c` (v9620)。
- `cmake -DGGML_VULKAN=ON` で **`build-vulkan/` に分離ビルド**（ROCm の `build/` には触れない）。
- 初回は `SPIRV-Headers` の CMake config が無くビルド不可 → `spirv-headers` / `spirv-tools`（LunarG リポジトリ、SDK 版一致）を導入して解決。
- ビルド成功。ROCm pin (v8533) では gfx900 で `unknown type name '__hip_fp8_e4m3'` となる master が、**Vulkan では HIP device コンパイルを行わないためそのまま通る**ことを確認。

## Phase 1: 起動・KV q8_0/FA 可否

- 4枚の RADV VEGA10 を検出（`GGML_VK_VISIBLE_DEVICES=0,1,2,3` で llvmpipe を除外）、ctx=131072 で起動・`/health` OK。
- **KV q8_0 + FA は RADV で正常動作**。
- **FA=0 + KV q8_0 は不可**: `llama_init_from_model: V cache quantization requires flash_attn`（CUDA/ROCm と同じ制約）。FA 無効化には f16 KV が必要。
- **prompt cache・context checkpoint は Vulkan でも機能**（`prompt cache is enabled, size limit: 8192 MiB` / `created context checkpoint` を確認、ROCm と同等）。
- **新 master 由来の注意**: v9620 は `--defrag-thold` を deprecated 化（ログに `DEPRECATED: --defrag-thold is deprecated and no longer necessary to specify`）。start.sh は全サーバ共通で `--defrag-thold 0.1` を渡すため Vulkan(新 master) では警告が出る（無害。将来 start.sh から除去可）。また v9620 はロード時に device メモリへ自動 fit する `-fit` 機構を持つ（FA=0 時は `common_fit_params` がエラーを返す）。

## Phase 2: ub 掃引 × FA 有無

KV q8_0 / ctx=131072 / 32k プロンプト（31972 トークン）で計測。VRAM は `rocm-smi`（amdgpu 経由、Vulkan 下でも有効）。

| ubatch (FA=1) | GPU[0] load VRAM | prompt t/s | eval t/s | 結果 |
|--------|-------------|-----------|----------|------|
| 2048 | 8.72GB | 401.4 | **15.23** | ✅ 安定 |
| 3072 | 8.72GB | 405.6 | 14.45 | ✅ 安定 |
| 4096 | 8.72GB | **405.3** | 14.76 | ✅ 安定（ROCm では OOM） |

- **ub は VRAM・prompt にほぼ無影響**（VRAM は GPU[0] 8.72GB で完全一定、prompt ~405 t/s 一定）。ROCm の「ub を上げると VRAM 肥大→OOM、prompt 低下」という挙動は **RADV では起きない**。
- eval は僅差で ub=2048 が最良（15.23 t/s）。VRAM・prompt が同一のため、**ub=2048 を採用**（ROCm 構成と一致し運用が単純）。
- **FA=0（q8_0）は全 ub で起動失敗**（上記 V cache quantization 制約）。FA を切るには f16 KV が必要だが、f16 トライアル中にホストがダウンしたため（核心発見サマリの注意参照）f16 系列は安全のため打ち切り。

## Phase 3: 128k 安定性

ub=2048 / FA=1 / q8_0 で **100k トークン（99853）プロンプト**を処理:

| 指標 | 値 |
|------|-----|
| prompt (100k) | **216.2 t/s**（ROCm の 90k ~60 t/s に対し約3.6倍）|
| eval | 12.6 t/s |
| 処理時間 | 465s（≈7.7分。ROCm の 90k ~25分に対し大幅短縮）|
| GPU[0] VRAM（処理中） | **8.72GB で一定**（32k 時と不変＝処理長に非依存）|
| ホスト安定性 | ALIVE・OOM なし |

- KV は ctx=131072 分をロード時に確保、FA workspace は ubatch 比例で一定のため、**プロンプト長を増やしても VRAM は増えない**（32k→100k で GPU[0] 8.72GB のまま）。16GiB（17.16GB）に対し余裕 ~8.4GB。
- prompt 速度は 32k の 405 t/s から 100k で 216 t/s へ低下するが（高 ctx での全プロンプト再処理特性）、ROCm より遥かに高い水準を維持。

## ROCm vs Vulkan 比較

| 項目 | ROCm/HIP | Vulkan (RADV) |
|------|----------|---------------|
| llama.cpp | **pin `0fac87b15` (v8533)** | **master `57fe1f07c` (v9620)・pin 不要** |
| バックエンド指定 | `-DGGML_HIP=ON -DAMDGPU_TARGETS=gfx900` | `-DGGML_VULKAN=ON`（アーキ指定不要）|
| prompt t/s (32k) | 122.8 | **~405（×3.3）** |
| prompt t/s (~90-100k) | ~60 | **216（×3.6）** |
| eval t/s (32k) | **24.5** | ~15（×0.6）|
| GPU[0] VRAM | 11.6〜16.8GB（ub 比例）| **8.72GB（ub 非依存）** |
| ub=4096 | **OOM クラッシュ** | 動作（VRAM 一定）|
| VRAM 分散 | GPU[0] 偏り | 均等 |
| 確定 ub | 2048（演算最適点）| 2048（ub 非依存のため踏襲）|

**使い分け指針**: 長文要約・RAG 等 **prefill 律速のワークロードは Vulkan が圧倒的に有利**。短文対話で生成速度を重視するなら ROCm が有利。pin 保守を避けたい・最新 master 機能を使いたい場合も Vulkan。

## 確定構成（start.sh 反映済み）

`start.sh` は `MI25_BACKEND=vulkan` 指定時に `build-vulkan/bin/llama-server` を使い、`GGML_VK_VISIBLE_DEVICES=0,1,2,3` を付与する。SERVER_OPTS は ROCm と共通:
```bash
SERVER_OPTS="--flash-attn 1 --poll 0 -b 2048 -ub 2048"   # KV q8_0 / ctx 131072
```

## 再現方法

1. **（前提）Vulkan ビルド依存**（未導入の場合、sudo）:
   ```bash
   ssh mi25 'sudo apt-get install -y spirv-headers spirv-tools'
   ```
2. **ロック取得 + Vulkan ビルド・起動**:
   ```bash
   .claude/skills/gpu-server/scripts/lock.sh mi25
   MI25_BACKEND=vulkan .claude/skills/llama-server/scripts/start.sh mi25 \
     "unsloth/Qwen3.6-35B-A3B-GGUF:UD-Q4_K_XL" 131072
   ```
   - 初回は master を取得し `build-vulkan/` をビルド（ROCm の `build/` は無傷）。
3. **検証**:
   ```bash
   curl -sf http://10.1.4.13:8000/health
   ssh mi25 'rocm-smi --showmeminfo vram | grep Used'   # GPU[0] ~8.7GB
   # 32k/100k プロンプトを /v1/chat/completions に送り timings の prompt/eval t/s を確認
   ```
4. **ub/FA の上書き探索**（任意）: `EXTRA_LLAMA_OPTS="--flash-attn 1 -ub 4096"` 等を付与。

## 既知の課題・今後

- **高負荷時のホストダウン**: f16 KV + FA トライアル中に mi25 がダウン（要電源再投入）。q8_0 構成では再現せず安定。f16 KV / RADV シェーダ経路の安定性、または高負荷 I/O が要因と推定。**本番は q8_0 構成に限定**。再発時の自動復旧のため iLO 認証情報の `power.sh setup` 登録が望ましい。
- **master 追従のリスク**: Vulkan は pin しないため、将来の master 変更で RADV ビルド/挙動が壊れる可能性がある（pin しない方針の代償）。ビルド失敗時は直近の動作コミットを一時 pin する運用も検討。
- **eval 速度**: Vulkan の生成速度は ROCm の約0.6倍。生成律速用途には ROCm を併用。
- **f16 KV 系列の未計測**: FA 有無の完全な比較（f16）はホストダウンのため打ち切り。安定性確認後に再開する場合は監視下で慎重に。
