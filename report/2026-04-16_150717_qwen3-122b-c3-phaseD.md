## 前提・目的

前身レポート Phase C で `numactl --interleave=all` (C-C2) が **eval 11.91 t/s（C-C1 比 +60.7%）** で採用された。しかし以下 3 件が未検証だった:

- **F-1**: コールドスタート時の interleave 均等分散（drop_caches 前提）
- **F-2**: `interleave=all` + `--numa distribute` 併用の相補効果
- **F-3**: `--threads 40` + `numactl -N1 -m1` での C-C1 再計測（オーバーサブスクリプション仮説の直接検証）

本作業ではこれらを検証し、C-C2 を上回る構成があれば採用候補とすることを目的とした。

### スコープ変更

計画時点で drop_caches を必要とする F-1 検証を予定していたが、llm ユーザーに `sudo sysctl` 権限がないため F-1 はスキップした（warm cache 前提で検証）。代わりに F-3 の発見を受けて以下 2 構成を追加計測した:

- **F-4**: `interleave=all` + `--threads 40` （両ノード 40 スレッド）: 「単一ノード効果」と「1:1 スレッドマッピング効果」の切り分け
- **F-5**: `numactl -N0 -m0` + `--threads 40` （Node 0 単一 40 スレッド）: Node 間対称性の確認

### 成功条件

- C-D2 / C-D3 いずれかが C-C2 中央値 11.91 × **1.03** 以上（+3%）→ 採用候補
- C-D3 が前身 Phase B=11.50 近傍（±10%）に戻ればオーバーサブスクリプション仮説が支持される

## ボトルネック分析

### C-D3 がなぜ C-C2 より 26% も速いのか

想定外の大発見だった。以下の 3 要因が複合:

1. **NUMA リモートアクセスが実質 0**: モデルファイルのページキャッシュが Node 1 に偏在（N0=0.01%）。`--cpunodebind=1 --membind=1` により全スレッドが Node 1 の CPU から Node 1 のメモリにアクセスするため、ノード間 QPI/UPI 通信が不要
2. **1:1 スレッド⇄CPU マッピング**: 40 スレッド × 40 論理 CPU で完全非オーバーサブスクリプション。voluntary_ctxt_switches=273 と極小
3. **SMT（HT）活用**: Node 1 の 40 論理 CPU = 20 物理コア × 2 ハイパースレッド。CPU 集約的な演算を物理コアに均等にスケジューリング

前身 Phase C 時点では「両ノード活用が本質」と結論されたが、本計測の C-D5（interleave + threads=40、両ノード 40 スレッド）が 10.94 t/s と C-D3 の 72.8% にとどまったことで、**両ノード活用はページキャッシュが Node 1 偏在している現状では不利**と判明した。

### C-D5 (interleave + threads=40) が C-D3 より遅い理由

- 40 スレッドが Cpus_allowed_list=0-79 に分散 → Linux スケジューラが両ノードにスレッドを配置
- メモリは interleave で物理的にはラウンドロビンだが、ページキャッシュが Node 1 偏在のため実効的には Node 1 のメモリに大半がある
- Node 0 のスレッドが Node 1 のメモリに**リモートアクセス** → モデル重みのロードがボトルネック
- 「両ノード 80 CPU を均等に使う」ケース（C-C2/C-D4）より、40 スレッドが広い CPU 範囲に分散される分、**スレッド間の距離が広がり cache affinity が悪化**

### C-D6 (Node 0 単一) が C-D3 より大幅に遅い理由

- モデルファイルのページキャッシュが Node 1 に残存（Node 0 には Heap 243 MiB と少量の Private のみ）
- Node 0 の 40 スレッドが Node 1 のメモリに**100% リモートアクセス**
- NUMA 距離のペナルティが直撃 → C-D3 の 71.2%（10.70 / 15.03）
- もし drop_caches 後にコールドスタートすれば Node 0 にページキャッシュが乗り、C-D3 に近い性能が出る可能性あり（未検証）

### `--numa distribute` が効かない理由（C-D2）

- llama.cpp の `--numa distribute` はスレッドを両ノードにバインドするが、メモリ配置を制御しない
- モデルファイルのページキャッシュが Node 1 偏在のため、Node 0 スレッドは依然としてリモートアクセス
- C-C2 (`interleave=all` のみ) と比べて **−2.5%**（11.61 vs 11.91）→ スレッドの明示バインドがむしろスケジューラに干渉し、わずかに悪化

### `--threads -1` と `--threads 80` 明示の差異（C-C2 vs C-D4）

- C-C2 (`--threads -1`) = 11.91 / C-D4 (`--threads 80`) = 11.88 → **ほぼ同一**
- `--threads -1` は `sysconf(_SC_NPROCESSORS_ONLN)` から 80 を決定することを裏付け
- Phase C の「C-C1 が -N1 -m1 で 80 スレッド起動」の推定は正しい
