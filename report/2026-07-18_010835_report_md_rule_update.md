# REPORT.md 更新 — 概要必須ルール導入とセクション名のカノニカル化

- **報告日時**: 2026年7月18日 01:20 JST

## 概要

このプロジェクトのレポート作成ルール（REPORT.md）を、より成熟した形にアップデートした。既存の REPORT.md は簡素な箇条書き列挙にとどまっており、冒頭の要約セクションの規定・タイトルの書き方の指針・メタ情報の形式・実運用で定着しているセクション構成といった要素が抜けていたため、実運用と乖離しつつあった。

参考にしたのは、別プロジェクトで運用されているより詳細な REPORT ルールを含む CLAUDE.md である。特に「冒頭に **概要** セクションを必ず置き、平易な日本語で通読できる文章にする」というルールが明文化されており、これを本プロジェクトにも取り込むことが本改訂の出発点となった。

進め方としては、まず参考ルールを読み込み、次に本プロジェクトの近時分のレポートを調査して、実運用で定着しているセクション名・タイトルの長さ・メタ情報の書式・添付ファイルの使い方・PNG 埋め込みの実施状況を確認した。その上で、参考ルールをそのまま適用すると本プロジェクトの実運用と衝突する部分（実運用では「核心発見サマリ」中心で「概要」を書くレポートが少ない、等）を整理し、衝突ポイントについてユーザーに意思決定を仰いだ。

その結果、次の 4 点を決定した。「概要」を新規レポートから必須とすること。「概要」の内部構造は段落形式とすること。複数の表記揺れがあるセクション名についてはカノニカル名を推奨として明示すること。そして「作成者」フィールドについては、当初は削除する方向で進めていたが、その後のユーザー指示を受けて方針を反転させ、レポートを書いた Claude のモデル名を記録するフィールドとして活用することとした、の 4 点である。

実装としては、レポート作成ルール本体を全面書き換えして上記 4 決定を反映するとともに、関連する別ファイルの該当箇所も表現を揃える形で微修正した。フォーマット例も、既存の簡素な例から実運用に即した長めの例へと差し替えた。

適用範囲としては、本ルールは新規レポートから適用し、既存レポート（「概要」セクションを持たないもの）については遡及的な書き換えは行わない。この方針は REPORT.md 本体にも明記した。

次のセッション以降、本プロジェクトで plan mode を経由したまとまった作業を行った場合、レポート冒頭に「概要」セクションを必ず設け、通読できる平易な日本語でレポート全容をまとめることになる。本レポート自体がその第一号となる。

## 変更内容

### REPORT.md（全面書き換え）

新設・改訂した節：

- **レポート作成タイミング**（新設）: plan mode を経由したまとまった作業では完了時にレポート必須
- **タイトル (H1)**（新設）: 50 字以内、識別子は最小限、日時はタイトルに入れない、em dash 形式可
- **メタ情報**（新設）: 実施日時 / 報告日時 / 作成者 の 3 行 bullet、括弧書き要約付き、作成者にはレポートを書いた Claude のモデル名を記載（例: `Claude Opus 4.7 (1M context)` / `Claude Fable 5`）
- **概要 (必須)**（新設）: 通読できる平易な日本語で 5〜8 段落、細かい事実は他セクションに委ねる、位置は H1 とメタ情報の直下（他セクションより先）
- **セクション一覧 (カノニカル名)**（新設）: 11 セクションを要否欄付きの表で列挙、表記揺れがある場合は別表現欄に併記。並び順は必須ではない旨も付記
- **核心発見サマリ の書式**（新設・既存ユーザーメモリ由来）: 実測系では PNG 埋め込み必須、コード変更のみは省略可
- **添付ファイル / プランファイルの添付**（既存ルールを継承しつつ明文化を強化）
- **フォーマット例**（改訂）: 実運用に即した長めの完全例に差し替え

### CLAUDE.md（1 行修正）

「重要な制約」表の「レポート作成」行について、REPORT.md 側と表現を揃え、「まとまった作業」の含意と「概要必須」への言及を追加：

```diff
-| レポート作成 | plan mode で計画を立てた場合は、**必ず**対になるレポートを作成すること（ユーザから明示的に不要と指示された場合を除く）。フォーマットは [REPORT.md](REPORT.md) に従う |
+| レポート作成 | plan mode で計画を立ててまとまった作業を行った場合は、完了時に**必ず**対になるレポートを作成すること（ユーザから明示的に不要と指示された場合を除く）。フォーマット・必須セクション（**概要**必須ほか）は [REPORT.md](REPORT.md) に従う |
```

## 意思決定の記録

以下の 4 点は、参考ルールと本プロジェクトの実運用（過去約 2 ヶ月の 27 件のレポート慣習）を照合した結果、ユーザーに確認して決定した：

1. **冒頭の要約セクション** → 「概要」を必須。「核心発見サマリ」は必要に応じて執筆（技術詳細アブスト用）。「概要」の位置づけは、ユーザー（人間）が手短にレポート全容を把握するためのもの
2. **「概要」の内部構造** → 段落形式（5〜8 段落）とし、細かい事実は他セクションに委ねる（4 サブセクション分割形式は採用せず）
3. **セクション名の表記揺れ** → カノニカル名を推奨として明示（既存レポートの書き換えは強制しない）
4. **「作成者」フィールド** → 当初はほぼ使われていない実運用に合わせて削除する方向で進めていたが、後続のユーザー指示を受けて方針を反転。**Claude のモデル名を記録するフィールド**として活用することとした（例: `Claude Opus 4.7 (1M context)`, `Claude Fable 5`）。モデル名はシステムプロンプトの「You are powered by the model named ...」で確認できる
