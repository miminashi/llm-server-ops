## 実行結果サマリ

### 1. Explore サブエージェントの先行調査結果 ✅

| 発見 | 箇所 | 意味 |
|---|---|---|
| **非VMM pool 量子化 = 256 B** | `ggml-cuda.cu:422` | `look_ahead_size = 256 * ((look_ahead_size + 255)/256)` |
| **VMM granularity = device 固有 (典型 2 MiB)** | `ggml-cuda.cu:L504-509` | `reserve_size = granularity * ((reserve_size + granularity - 1) / granularity)` |
| **CUDA tensor alignment = 128 B** | `ggml-cuda.cu:772-776` | `ggml_backend_cuda_buffer_type_get_alignment() return 128;` |
| **chunk max_size 更新 (graph allocator)** | `ggml-alloc.c:303` | `chunk->max_size = MAX(chunk->max_size, offset + size);` |
| **pool size 集計** | `ggml-alloc.c:916-919` | `new_size += new_chunk_size;` |

**重要**: allocator には「1 MiB 境界」は存在しない ⇒ **Phase Sb-src 候補 D の形式棄却に直結**

### 2. 数値シミュレーション（alloc_sim.py）による候補 D 否定 ✅

9 SSM × `(ub+128) × 8192 × 4 bytes` per layer を 3 量子化単位で計算:

| 量子化単位 | ub=1585 合計 | ub=1586 合計 | Δ (1585→1586) | 評価 |
|---|---|---|---|---|
| 128 B (tensor align) | 481.781 MiB | 482.062 MiB | +0.2812 MiB | 連続線形、step なし |
| 256 B (非VMM pool) | 481.781 MiB | 482.062 MiB | +0.2812 MiB | 同上 |
| 2 MiB (VMM granularity) | 486.000 MiB | 486.000 MiB | **+0.0000 MiB** | 大 block 平坦、ub=1664 で +18 MiB 単 step |

いずれの量子化単位でも **Phase Sb-fine3 の観測 (1585→1586 で slope が 0→0.2853 MiB/tok へ切り替わる step)** は再現できない。
⇒ **候補 D 棄却確定**。

### 5. 最有力仮説 J: 9 層 SSM × VMM block 非同期累積 ⭐

9 SSM 層の fused GDN 出力は各層が独立に VMM 2 MiB block を取得。各層が境界を跨ぐ ub は:

```
per-layer raw = 4 × 8192 × (ub + 128) bytes
2 MiB 境界 = 2,097,152 bytes
ub + 128 ≡ 0 (mod 64) のとき per-layer size が 2 MiB の整数倍
= ub = 64k - 128 (k ∈ ℤ)
  k=26 → ub=1536, k=27 → ub=1600, k=28 → ub=1664, k=29 → ub=1728
```

一方、9 層それぞれが別個に VMM block を占有する場合:
- 9 層の **alignment padding の累積**が ub=1585→1586 で「内部フラグメント」を 1 block 分吸収
- これにより「平坦域→線形域」の切替えが出現

**本仮説は「VMM granularity の非同期累積」説であり、単純な 2 MiB 跨ぎ仮説と違い、9 層配置の細部と GPU allocator の実装依存 detail が絡む**。次 Phase で debug 出力による per-node size dump が必要。

### 6. FA tile の parallel_blocks（候補 I-b）補足 ✅

`fattn-common.cuh:1052-1125`:
- `max_blocks_per_sm` = occupancy から決まる定数 (P100 固定)
- `parallel_blocks` は efficiency optimizer で `ntiles_KV` まで拡大試行
- `parallel_blocks > 1` で `dst_tmp` / `dst_tmp_meta` を `cuda_pool_alloc` から動的確保

**注意**: `dst_tmp` は runtime allocation であり **compute buffer reservation (graph_reserve) には含まれない可能性が高い**。ただし graph_reserve 時にも同じ code path が走る場合は反映されうる。次 Phase で要検証。

## ボトルネック・副次発見の分析

### 1. 「allocator 仮説が全体的に単純すぎる」という知見

Phase Sb-src で提示された候補 D は「1 MiB pool 量子化」だったが、実際の allocator レイヤには:
- 128 B (tensor align)
- 256 B (非VMM pool)
- device granularity (VMM、典型 2 MiB)

の 3 段階しかなく、「1 MiB」という量子化単位は**どこにも存在しない**。Phase Sb-src の「482 MiB 境界」は観測と理論の 0.6 tok 差を埋めるための後付け仮説に過ぎなかった。

### 2. step の真因候補が絞り込まれた

graph nodes/splits 不変 + allocator 単純量子化否定により、候補は以下に絞られる:

- **候補 I-c**: build_graph 内で n_tokens 依存の tensor reshape が離散的に切り替わる
- **候補 J**: 9 層 SSM × VMM granularity の非同期累積によるマルチ block 境界

いずれも **debug ビルド + per-node tensor size dump** が最終確定に必要。

### 3. CUDA3 は純線形 (step なし) の確認

startup_logs 再分析で CUDA3 = 0.98 MiB/tok 純比例を追確認（1557.14 → 1558.12 → 1560.09 → 1564.02）。Phase R で「CUDA3 が ctx 完全不依存」だった観測とも整合。

### 4. Phase Sb-src の採用判定の再評価

| Phase Sb-src の判定 | 本 Phase での更新 |
|---|---|
| 候補 D = 最有力（0.6 tok 誤差） | **棄却**（1 MiB 境界は alloc に存在せず） |
| slope 0.2853 = GDN 出力 98.6% | **維持**（変更なし） |
| 境界 ub\*=1586 の根拠 = 不完全 | **真因不明、候補 J/I-c に絞り込み** |

## 採用判定

| 項目 | 結果 |
|---|---|
| ggml-alloc.c の具体量子化単位特定 | ✅ 128 B / 256 B / device granularity (2 MiB) |
| 候補 D の数値検証による棄却 | ✅ 3 量子化単位全てで step 不発生 |
| 候補 E/G/H の棄却 | ✅ コード/ログ両面で否定 |
| 候補 J の提示と形式化 | ✅ 「9 層 × VMM block 非同期累積」として定式化 |

| GPU ロック未取得 | ✅ 読み取り専用、競合なし |

**結論**: **Phase Sb-alloc は主目的を達成**。Phase Sb-src の候補 D は allocator レイヤ由来ではないことを確定し、真因の方向性を **候補 J (9 層 × VMM) / 候補 I-c (build_graph 離散)** に絞った。次 Phase は debug ビルドによる per-node dump が最短 path。

## 確定モデル（更新版）

**Phase Sb-src の物理モデルは不変**:

```
CUDA0 slope ≈ 4 × S_v × H × n_ssm_per_gpu bytes/tok = 0.28125 MiB/tok (主因、GDN 出力線形項)
             + 0.00405 MiB/tok (残差、FA dst_tmp / padding / 他 GPU 共有)
           = 0.28530 MiB/tok
```

**境界位置の根拠は本 Phase で再設定**:

- Phase Sb-src では「1 MiB allocator pool 量子化」と推定
- 本 Phase で「1 MiB 境界は allocator に存在せず」を確定
- 真因は **9 SSM × VMM granularity の非同期累積** または **build_graph 内の ub 依存離散処理** に移行

## 補足

### Phase Sb-alloc の核心発見（サマリ）

1. **allocator には「1 MiB 境界」は存在しない**:
   - 量子化単位は 128 B (tensor align) / 256 B (非VMM) / device granularity (VMM、典型 2 MiB) の 3 段階のみ
   - Phase Sb-src の候補 D は**棄却**

2. **VMM=yes 確認**:
   - t120h-p100 全 4 GPU で VMM 有効
   - granularity-based 量子化 path が active
   - 典型 granularity は 2 MiB (next Phase で実測確認推奨)

3. **graph nodes/splits は ub に完全不変**:
   - nodes=4473, splits=136 (pp) / 77 (tg) が ub=1585/1586/1588/1592 で固定
   - **候補 H (graph splits 境界) 棄却**
   - ⇒ step は「既存 node の shape 微変化」の累積

4. **候補 E (GDN kernel tile) も棄却**:
   - grid = (H, n_seqs, ceil(S_v/4)) = (64, 1, 32) は n_tokens 完全非依存
   - block = (32, 4, 1) も同様
   - n_tokens は kernel 内の for ループで処理、tile 境界なし

5. **残る最有力仮説**:
   - **候補 J**: 9 層 SSM × VMM 2 MiB granularity の **非同期累積**
     - 各層独立 block で配置 → alignment padding の累積が境界挙動を生む
   - **候補 I-c**: build_graph 内の ub 依存離散処理
     - qwen35moe.cpp の MoE router / attention tensor の離散 reshape

6. **次 Phase 最優先**:
   - **Phase Sb-tensor-dump** (debug build 必要、2-3 時間): per-node byte dump で候補 J vs I-c を確定
   - **Phase Sb-ctx-boundary** (GPU ロック、1.5 時間): 候補 J の ctx 非依存性検証
   - **Phase Sb-fa0** (GPU ロック、1 時間): 候補 J の fa 非依存性検証

### Phase Sb-src との対照

Phase Sb-src は「**slope 由来を確定（98.6%）し、境界位置を最有力仮説 (候補 D) に絞り込んだ**」ソース解析 Phase。本 Phase Sb-alloc はその候補 D を**数値とコードで否定**し、真因の方向性を更新する「検証 Phase」。両者の関係:
